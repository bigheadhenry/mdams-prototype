# 开发与工作流指南

## 1. 当前工作方式

MDAMS 目前采用的是“本地开发 + 测试验证 + 工作日志 + GitHub 同步”的方式。

核心原则：

- 先在本地把功能做完并验证
- 每次有改动都追加到 `docs/WORK_LOG.md`
- 关键文档要同步更新
- 最后再把变更推到 GitHub

## 2. 日常开发流程

### 2.1 修改代码或文档

建议在本地直接修改对应模块，然后立刻补充：

- 相关测试
- 工作日志
- 必要的 README 或文档说明

### 2.2 运行验证

建议至少跑：

- 后端 pytest
- 前端 build
- 前端 Playwright 回归

### 2.3 记录变更

任何影响功能、配置、部署、权限、接口、文档的变更，都需要写入 `docs/WORK_LOG.md`。

## 3. GitHub 同步

当当前轮工作完成后：

1. 确认工作区只包含预期变更
2. 提交本地 commit
3. 推送到 GitHub 分支
4. 如有需要更新 PR 说明

当前仓库已经习惯用 GitHub 分支和 PR 来承接阶段性工作。

## 4. 测试要求

建议把测试分成三层：

- **单元 / 契约测试**：验证函数、服务和 schema
- **集成 / Smoke 测试**：验证接口和主链路
- **前端回归测试**：验证登录态、菜单可见性和关键页面

## 5. 开发顺序建议

建议优先顺序如下：

1. 主链路功能
2. 权限与范围
3. 统一平台
4. 三维与展示
5. 测试和文档

## 6. 与工作日志的关系

工作日志不是附属品，而是项目的一部分：

- 它记录了做过什么
- 它帮助我们判断当前状态
- 它也帮助后面的人快速接手

## 7. 当前建议

如果以后新增功能，建议优先做到：

- 文档同步更新
- 测试同步补齐
- 日志同步记录
