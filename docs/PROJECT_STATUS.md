# MDAMS 项目状态与路线图

## 1. 项目定位

MDAMS Prototype 是一个面向博物馆馆内业务的数字资源管理原型。当前项目已经不再只是“二维影像 PoC”，而是逐步形成了以下几个稳定模块：

- 二维影像子系统
- 三维数据管理子系统
- 统一平台入口与统一资源目录
- 馆内登录、角色与权限框架
- 资源申请、审批与交付流程
- 测试分层与工作日志机制

当前阶段最合适的定位是：

> **一个可持续开发、可稳定演示的馆内数字资源底座，而不是完整生产级 DAMS。**

它已经验证了核心技术路线可行，但仍然需要继续把业务边界、权限模型、长期保存和平台化能力做得更稳。

## 2. 当前已经实现的能力

### 2.1 二维影像子系统

- 基础文件上传与资源列表
- SIP 风格入库与 fixity 校验
- PSB / BigTIFF 处理
- IIIF Manifest 生成
- Mirador 预览
- 原文件与 BagIt ZIP 下载
- 申请车、申请单、审批与交付包导出
- 资源可见范围与藏品责任范围控制

### 2.2 三维数据管理子系统

- 三维资源对象与多文件资源包管理
- 模型、点云、倾斜摄影图像分角色保存
- 版本号与 Web 展示状态管理
- 对象级聚合视图与对象级概览
- 三维 Web 查看器
- 藏品对象关联与生产链路记录
- 长期保存状态与展示状态分离

### 2.3 统一平台

- 统一资源目录
- 统一详情页
- 来源注册表与适配器
- 二维 / 三维来源接入
- 按 profile、状态、资源类型、可见范围筛选

### 2.4 权限与登录

- 用户、角色、会话、登录上下文
- 前端菜单按角色可见
- 后端关键接口按权限保护
- `RBAC + scope` 权限模型
- `collection_owner` 藏品责任范围过滤

### 2.5 测试与日志

- 后端 pytest 分层
- 前端 Playwright 登录态回归
- 三维元数据字典契约测试
- 三维生产链路测试
- 工作日志机制

## 3. 目前仍然存在的主要缺口

### 3.1 统一平台层还需要继续加强

当前已经有统一目录和统一详情，但平台层后面还可以继续补充：

- 更正式的统一检索索引
- 更多来源系统接入
- 更完整的资源聚合视图
- 更清晰的跨系统详情组织方式

### 3.2 权限模型还可以继续细化

目前已经有登录、角色和范围控制，但后续还需要继续补：

- 更真实的用户数据接入
- 责任范围的管理页面
- 审批与导出规则的进一步细化
- 更完整的审计和记录能力

### 3.3 长期保存与治理能力还不够完整

当前项目已经能跑通核心链路，但长期保存、迁移、校验、监控、告警等治理能力还需要持续补齐。

## 4. 当前阶段判断

目前更准确的判断是：

> **MDAMS 已经从单一技术原型推进到“稳定的馆内数字资源底座”阶段。**

这意味着：

- 关键链路已经可用
- 子系统边界已经开始清晰
- 统一平台方向已经成立
- 权限与利用流程已经开始具备业务语义

但它还不是一个完整产品化 DAMS，后续还需要继续做平台治理、统一检索和长期保存。

## 5. 下一步建议

下一步建议继续围绕以下三条主线推进：

1. **继续完善统一平台层**
   - 统一检索
   - 来源接入
   - 统一详情组织

2. **继续完善二维 / 三维子系统**
   - 二维资源继续稳固利用申请与展示链路
   - 三维资源继续补兼容性、查看器和保存能力

3. **继续补齐权限、治理与长期保存能力**
   - 更真实的用户与责任范围
   - 审计与审批
   - 长期保存与迁移
