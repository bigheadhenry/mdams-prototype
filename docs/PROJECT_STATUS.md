# MDAMS Prototype 项目现状与开发路线图

## 1. 项目定位

**MDAMS Prototype**（Museum Digital Asset Management System Prototype）是一个面向博物馆/展览数字资源场景的原型系统，重点验证以下能力：

- 数字资产上传、入库与基础管理
- 超大图像（尤其 PSB / BigTIFF）的处理与预览
- 基于 IIIF 的标准化图像服务与展示
- 面向家庭实验室环境的容器化部署可行性

当前阶段更适合定义为：

> **以数字资产入库、超大图像处理和 IIIF 预览为核心的技术原型。**

它已经具备一条可工作的主链路，但尚未完成完整的业务化 DAMS 功能建设。

---

## 2. 当前已实现能力

### 2.1 后端能力

当前后端基于 **FastAPI + SQLAlchemy + PostgreSQL + Celery + Redis**，已实现的核心能力包括：

- 基础文件上传
- SIP 风格入库接口
- 文件 SHA256 fixity 校验
- 资产列表查询
- 资产删除
- IIIF Manifest 动态生成
- BagIt ZIP 打包下载
- PSB 异步转换 BigTIFF
- 图像基础元数据提取

### 2.2 前端能力

当前前端基于 **React 18 + Vite + TypeScript + Ant Design + Mirador**，已实现：

- 基础仪表盘页面
- 资产表格浏览
- 文件上传
- 删除与下载操作
- Mirador 预览集成
- 客户端入库演示流程（含 hash / manifest / fixity 验证）

### 2.3 部署能力

当前仓库已经包含较完整的容器化部署结构，涉及：

- frontend
- backend
- celery_worker
- db
- redis
- cantaloupe
- filebrowser

部署设计已明显针对实际实验环境进行适配，包括：

- Linux + Docker 环境
- NAS / NFS 混合存储
- N100 低功耗服务器场景
- IIIF 图像服务链路

---

## 3. 当前项目判断

## 当前项目最成熟的部分

1. **数字资产入库主链路**
   - 上传
   - fixity 校验
   - 元数据提取
   - 入库记录

2. **超大图像处理链路**
   - PSB 上传
   - Celery 异步转换 BigTIFF
   - Cantaloupe 提供 IIIF 图像服务
   - Mirador 进行前端预览

3. **IIIF Manifest 动态生成**
   - 已具备标准化展示能力的原型基础

4. **面向真实环境的部署结构**
   - 不只是 demo，而是已经考虑你的实验室服务器/NAS 约束

### 当前项目的主要阶段结论

当前系统更准确地说是：

> **“高价值技术原型”已形成，但尚未进入完整业务产品化阶段。**

也就是说，它已经证明：

- 关键技术方案可行
- 核心链路可跑通
- 后续可以持续迭代

但尚未完全覆盖文档中设想的全部业务能力。

## 4. 当前主要缺口

### 4.1 命名与文档口径尚未完全统一

历史文档中存在多个名称并行：

- MEAM
- MDAMS
- Prototype
- Museum Enterprise Asset Management

本轮整理后，建议统一主口径为：

> **MDAMS Prototype**

完整展开为：

> **Museum Digital Asset Management System Prototype**

### 4.2 数据模型仍偏原型化

目前核心数据模型仍以 `Asset` 为主，尚不足以完整支撑：

- 编目体系
- 资产版本关系
- 衍生文件关系
- 利用申请/交付
- 收藏夹/购物车
- 用户与权限控制

### 4.3 前端功能仍以演示链路为主

当前前端实现重点集中在：

- 上传
- 浏览
- 预览
- 下载
- ingest PoC

而以下业务模块仍未完成：

- 高级检索
- 资产详情页
- 收藏夹
- 购物车
- 利用单流程
- 用户管理

### 4.4 后端结构开始需要模块化整理

随着功能扩展，当前接口分布将逐渐不利于维护。下一阶段应考虑进一步模块化拆分，例如：

- `routers/assets.py`
- `routers/ingest.py`
- `routers/iiif.py`
- `routers/downloads.py`

### 4.5 运行配置需要逐步抽离

当前项目中存在与部署环境绑定较强的配置，如：

- 局域网 IP
- NAS 路径
- 容器 URL
- 数据库/Redis 连接地址

下一阶段建议逐步迁移到更清晰的环境配置方式（如 `.env` + compose 环境变量）。

## 7. 当前推荐工作方向

如果只选一个当前最合适的方向，建议是：

> **先把项目从“能跑”整理成“可持续开发”的状态。**

这意味着下一步优先关注：

- 命名与文档统一
- 配置与结构整理
- 演示链路稳定化

在此基础上，再进入业务模块扩展，会更稳、更省成本。
