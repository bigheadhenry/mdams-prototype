# MDAMS Prototype 项目现状与开发路线图

## 1. 项目定位

**MDAMS Prototype**（Museum Digital Asset Management System Prototype）是一个面向博物馆/展览数字资源场景的原型系统，重点验证以下能力：

- 数字资产上传、入库与基础管理
- 超大图像（尤其 PSB / BigTIFF）的处理与预览
- 基于 IIIF 的标准化图像服务与展示
- 面向家庭实验室环境的容器化部署可行性

当前阶段更适合定义为：

> **以数字资产入库、超大图像处理和 IIIF 预览为核心的技术原型。**

它已经具备一条可工作的主链路，但尚未完成完整的业务化 DAMS 功能建设。

---

## 2. 当前已实现能力

### 2.1 后端能力

当前后端基于 **FastAPI + SQLAlchemy + PostgreSQL + Celery + Redis**，已实现的核心能力包括：

- 基础文件上传
- SIP 风格入库接口
- 文件 SHA256 fixity 校验
- 资产列表查询
- 资产删除
- IIIF Manifest 动态生成
- BagIt ZIP 打包下载
- PSB 异步转换 BigTIFF
- 图像基础元数据提取

### 2.2 前端能力

当前前端基于 **React 18 + Vite + TypeScript + Ant Design + Mirador**，已实现：

- 基础仪表盘页面
- 资产表格浏览
- 文件上传
- 删除与下载操作
- Mirador 预览集成
- 客户端入库演示流程（含 hash / manifest / fixity 验证）

### 2.3 部署能力

当前仓库已经包含较完整的容器化部署结构，涉及：

- frontend
- backend
- celery_worker
- db
- redis
- cantaloupe
- filebrowser

部署设计已明显针对实际实验环境进行适配，包括：

- Linux + Docker 环境
- NAS / NFS 混合存储
- N100 低功耗服务器场景
- IIIF 图像服务链路

---

## 3. 当前项目判断

## 当前项目最成熟的部分

1. **数字资产入库主链路**
   - 上传
   - fixity 校验
   - 元数据提取
   - 入库记录

2. **超大图像处理链路**
   - PSB 上传
   - Celery 异步转换 BigTIFF
   - Cantaloupe 提供 IIIF 图像服务
   - Mirador 进行前端预览

3. **IIIF Manifest 动态生成**
   - 已具备标准化展示能力的原型基础

4. **面向真实环境的部署结构**
   - 不只是 demo，而是已经考虑你的实验室服务器/NAS 约束

### 当前项目的主要阶段结论

当前系统更准确地说是：

> **“高价值技术原型”已形成，但尚未进入完整业务产品化阶段。**

也就是说，它已经证明：

- 关键技术方案可行
- 核心链路可跑通
- 后续可以持续迭代

但尚未完全覆盖文档中设想的全部业务能力。

## 4. 当前主要缺口

### 4.1 命名与文档口径尚未完全统一

历史文档中存在多个名称并行：

- MEAM
- MDAMS
- Prototype
- Museum Enterprise Asset Management

本轮整理后，建议统一主口径为：

> **MDAMS Prototype**

完整展开为：

> **Museum Digital Asset Management System Prototype**

### 4.2 数据模型仍偏原型化

目前核心数据模型仍以 `Asset` 为主，尚不足以完整支撑：

- 编目体系
- 资产版本关系
- 衍生文件关系
- 利用申请/交付
- 收藏夹/购物车
- 用户与权限控制

### 4.3 前端功能仍以演示链路为主，但第一阶段对象闭环已落地

当前前端实现重点集中在：

- 上传
- 浏览
- 预览
- 下载
- ingest PoC
- 对象详情页（第一阶段）

其中，面向**单件二维文物图片资源对象**的第一阶段闭环已在现网完成最小落地：

- 上传或入库后可围绕对象记录展开；
- 已新增对象详情视图，集中展示对象摘要、文件信息、技术元数据、处理状态；
- 已提供更清晰的访问与输出入口，包括 IIIF Manifest、Mirador 预览、原文件下载、BagIt 包下载；
- 处理状态表达已从“仅能否成功”提升为可读提示，例如“等待处理”“处理完成，可预览”。

但以下业务模块仍未完成：

- 高级检索
- 收藏夹
- 购物车
- 利用单流程
- 用户管理

### 4.4 后端结构开始需要模块化整理

随着功能扩展，当前接口分布将逐渐不利于维护。下一阶段应考虑进一步模块化拆分，例如：

- `routers/assets.py`
- `routers/ingest.py`
- `routers/iiif.py`
- `routers/downloads.py`

### 4.5 运行配置需要逐步抽离

当前项目中存在与部署环境绑定较强的配置，如：

- 局域网 IP
- NAS 路径
- 容器 URL
- 数据库/Redis 连接地址

下一阶段建议逐步迁移到更清晰的环境配置方式（如 `.env` + compose 环境变量）。

如果只选一个当前最合适的方向，建议是：

> **先把项目从“能跑”整理成“可持续开发”的状态。**

- 现网 `worker` 已切换到与新字段兼容的新代码；

- 数据库已补齐 `resource_type` 与 `process_message` 字段，使现网数据结构与新代码一致；
- 通过真实上传样本，成功生成新对象 `asset 44`；
- `asset 44` 的详情接口、原文件下载、BagIt 下载均返回成功；
- 已确认对象详情返回 `resource_type = image_2d_cultural_object`，并能显示 `process_message = 处理完成，可预览`。

### 8.2 阶段判断更新

因此，当前 MDAMS Prototype 不再只是“上传 / 列表 / 预览 / 下载的技术能力集合”，而已经进入：

> **单件二维文物图片资源对象闭环原型（第一阶段，已现网验证）**

这意味着当前最值得继续推进的方向，已经不再是“是否能做出对象表达”，而是：

- 如何把当前对象详情页进一步泛化为更稳定的对象表达层；
- 如何让不同资源类型逐步进入共同框架；
- 如何在不破坏现有演示闭环的前提下继续增强状态、关系与输出能力。

### 8.3 2026-03-14 用户异机访问补充结论

用户已在同网络的另一台电脑上完成真实访问测试，结果表明：

- 系统整体访问与主链路操作基本正常；
- 除“部分图片在 Mirador 中打开失败”外，未发现其他明显问题；
- 因而当前阶段可以认为：**第二轮对象结构化详情增强已基本稳定落地**；
- Mirador 的部分图片兼容性问题暂作为后续独立排查专题保留，不阻断本阶段收口。

### 8.4 2026-03-14 第三轮阶段更新：对象处理轨迹已上线

在第二轮已完成“对象结构 / 访问路径 / 输出动作”整理的基础上，第三轮进一步为对象详情页增加了最小“处理轨迹 / 生命周期记录”能力。

本轮已确认：

- 现网 `GET /assets/{asset_id}` 已返回 `process_timeline`；
- 当前最小轨迹已覆盖：对象建立、上传 / 入库完成、Fixity 记录、基础元数据形成、访问主文件准备、预览就绪、输出动作具备；
- 现网前端已加载第三轮静态资源，用户入口 API 已能返回完整轨迹结构；
- 因而当前对象详情页已从“对象静态结构页”进一步提升为：

> **对象结构 + 最小生命周期轨迹** 的资源对象页

这说明当前 MDAMS 第一阶段的对象表达，已经不再只停留在“能访问、能下载”，而开始具备更接近真实资源对象管理界面的过程层表达。
