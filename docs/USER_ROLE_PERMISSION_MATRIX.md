# MDAMS 用户类型与权限矩阵

## 目的

本文档用于定义 MDAMS 当前阶段的用户类型、角色职责、资源可见范围与权限边界。

当前系统定位为馆内业务系统，不面向公众开放，因此本文档不包含公众访客角色。

## 设计原则

1. 角色与用户分离  
用户可以同时拥有多个角色，角色代表职责，不直接等同于现实中的岗位编制。

2. 二维与三维分别授权  
二维影像与三维资源需要分别控制权限，避免一个全局角色自动获得所有资源域的操作权。

3. 可见范围与业务状态分离  
资源的流程状态不等于资源的可见范围。后续应分别维护 `workflow_status` 与 `visibility_scope`。

4. 藏品责任范围优先  
涉及藏品责任人时，权限判断不仅取决于角色，也取决于资源是否属于该用户负责的藏品范围。

5. 前后端同时生效  
前端负责菜单和页面裁剪，后端负责接口和资源访问控制，不能只做前端隐藏。

## 用户类型

当前系统分为两大类使用者：

- 数字化流程人员
- 业务人员

## 角色定义

### 数字化流程人员

#### 1. 二维结构化数据录入 `image_structured_editor`

- 负责二维资源的结构化元数据录入与修订
- 典型内容包括文物号、文物名称、分类、对象关联等

#### 2. 二维非结构化处理/上传 `image_ingest_operator`

- 负责二维文件上传、技术元数据补录、衍生资源处理与入库准备

#### 3. 二维入库审核 `image_ingest_reviewer`

- 负责审核二维资源是否满足正式入库条件
- 不负责利用申请审批

#### 4. 二维资源管理/元数据维护 `image_resource_manager`

- 负责二维资源的日常维护、对象关联调整、元数据修订与质量维护

#### 5. 三维处理/上传 `three_d_operator`

- 负责三维对象录入、文件包上传、版本维护、Web 展示版选择与技术资料维护

#### 6. 数据申请审批 `application_reviewer`

- 负责二维资源利用申请的审批、拒绝、交付导出
- 不负责资源入库审核

#### 7. 系统管理 `system_admin`

- 负责用户、角色、字典、系统配置、来源接入和全局治理

### 业务人员

#### 8. 藏品管理者 `collection_owner`

- 可管理自己负责藏品对应的数字资源元数据
- 可浏览全馆开放的数字资源
- 可浏览自己责任范围内的受限资源

#### 9. 数据浏览/申请人员 `resource_user`

- 可浏览开放资源
- 根据浏览结果决定是否发起数据申请
- 不具备审批权

## 资源可见范围

当前第一版建议只定义两级可见范围：

- `open`  
  馆内开放浏览的资源，具备浏览权限的馆内用户均可查看

- `owner_only`  
  仅相关责任范围内人员可见的资源，例如对应藏品管理者及被授权岗位

## 业务状态建议

后续应将资源流程状态独立维护，避免与权限混用。第一版建议：

- `draft`
- `pending_ingest_review`
- `ingested`
- `published`
- `frozen`

## 角色说明与范围补充

### collection_owner

- 可查看所有 `open` 资源
- 可查看自己负责藏品对应的 `owner_only` 资源
- 可编辑自己负责藏品对应资源的业务元数据
- 不具备申请审批和系统管理权限

### resource_user

- 浏览与申请属于同一类人
- 可以先浏览开放资源，再决定是否提交申请
- 后续前端应展示“我的申请”而非审批入口

### image_ingest_reviewer 与 application_reviewer

- 两者是不同角色
- 实际业务中可以由同一名用户同时持有这两个角色
- 系统层面不应把二者强行合并

### image_ingest_reviewer

- 二维资源

### image_resource_manager

### three_d_operator

- 三维管理

### application_reviewer

- 申请管理
- 二维资源浏览

### collection_owner

- 二维资源浏览

## 后端权限实现建议

第一阶段建议采用 `RBAC + 资源范围判断`：

- RBAC 负责判断是否具备某类动作权限
- 资源范围判断负责限制是否只能访问 `open` 或 `own_scope`

建议的权限命名方式：

- `image.view`
- `image.edit`
- `image.upload`
- `image.ingest_review`
- `three_d.view`
- `three_d.edit`
- `three_d.upload`
- `application.create`
- `application.approve`
- `application.export`
- `system.manage`

范围型补充判断：

- `scope = all`
- `scope = open`
- `scope = own_scope`

## 第一阶段落地建议

1. 先引入当前用户上下文与角色模拟数据  
用于前端按角色裁剪菜单和页面入口。

2. 再抽象权限判断函数  
例如前端 `can()`，后端 `require_permission()`。

3. 先保护关键流程  
优先保护：

- 二维上传与删除
- 三维上传与版本维护
- 申请审批与交付导出
- 藏品责任范围下的元数据编辑

4. 最后再接真实登录体系  
在角色与权限边界稳定之前，不建议先把登录方案做重。

## 当前结论

MDAMS 当前最合适的权限框架是：

- 以馆内业务用户为中心
- 以角色职责为主
- 以二维和三维资源域分别授权
- 以藏品责任范围作为资源级补充约束
- 以 `RBAC + scope` 作为第一阶段实现方案
