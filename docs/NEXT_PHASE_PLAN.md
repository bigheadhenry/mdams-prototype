# MDAMS Prototype 下一阶段计划

## 1. 目标

这份文档用于描述 MDAMS Prototype 接下来一段时间最值得优先推进的方向。

当前系统已经不只是一个二维影像 PoC，而是已经具备：

- 二维影像资源管理与 IIIF 预览
- 三维资源管理与 Web 查看
- 统一平台目录与统一详情
- 登录、角色、权限与范围控制
- 利用申请与交付闭环

因此，下一阶段的目标不再是“证明技术可行”，而是：

> **把当前可用能力整理成稳定、可维护、可逐步扩展的馆内数字资源底座。**

## 2. 当前阶段判断

### 2.1 已经完成的部分

当前项目已经完成的重点包括：

- 二维影像主链路可用
- 三维子系统可用
- 统一平台入口可用
- 登录和权限框架可用
- 申请与交付流程可用
- 工作日志和测试分层已建立

### 2.2 仍然需要继续补的部分

当前最需要继续补的不是“更多零散功能”，而是：

- 更稳的统一平台层
- 更清晰的资源范围与权限规则
- 更完整的治理和长期保存能力
- 更实用的测试和回归体系

### 2.3 当前阶段的准确定位

当前最准确的阶段定义是：

> **高价值技术原型已经成型，正在向稳定可持续开发底座过渡。**

## 3. 下一阶段总目标

下一阶段建议拆成三条主线：

1. **固化核心主链路**
   - 让上传、入库、预览、下载、申请、交付都更稳定

2. **增强工程可维护性**
   - 让项目更容易理解、排错、回归和扩展

3. **继续补齐最小业务表达**
   - 让系统看起来更像一个完整的业务系统，而不是技术拼装

## 4. 优先级建议

### P0：主链路稳定化

目标：

> 把当前最有价值的链路做成标准、稳定、可验证的主流程。

建议任务：

- 统一 README 和项目说明
- 明确当前已实现与未实现的范围
- 梳理后端路由结构
- 统一关键配置入口
- 形成标准演示流
- 补充错误处理和状态反馈

### P1：工程化与可维护性提升

目标：

> 让项目更容易运行、排错和继续开发。

建议任务：

- 继续拆分后端模块
- 补充更完整的测试分层
- 强化登录后的回归测试
- 继续完善工作日志和状态文档
- 收敛配置项和环境变量说明

### P2：最小业务模型补齐

目标：

> 让系统从“技术可行”变成“业务可理解”。

建议任务：

- 资源对象与藏品对象关系进一步完善
- 二维 / 三维资源的管理语义统一
- 申请、审批、交付的状态再梳理
- 统一资源目录和统一详情继续增强

### P2 补充：缩略图规格与生成策略

目标：

> 为二维影像列表页和快速浏览补齐一套稳定的缩略图规范与生成方式。

建议任务：

- 定义缩略图尺寸、格式、质量和命名规则
- 明确哪些资源在上传后需要预生成缩略图
- 明确哪些资源继续走即时切片 / 即时缩放
- 明确列表页、统一目录页、申请页分别读取哪一层图片
- 明确缩略图失败时的降级策略
- 明确缩略图与原图、衍生图、IIIF 图像服务之间的关系

这项工作建议放在当前主链路稳定之后推进，不要和主入库链路混在一起做。

### P3：扩展能力预留

目标：

> 给未来的检索、AI、编目模板、权限细化和更多来源接入留出空间。

建议任务：

- 统一检索索引
- 更多来源系统接入
- 权限粒度细化
- 编目模板和标准字典
- OCR / AI 标签
- 更完整的审计和治理能力

## 5. 建议的下一阶段路线

### 路线 A：先把当前主链路做稳

这是最优先的路线。

重点包括：

- 上传与入库
- IIIF 与 Mirador
- 申请与交付
- 三维查看器
- 统一平台目录
- 权限与范围控制

### 路线 B：继续完善统一平台

在主链路稳定后，继续补：

- 统一检索
- 来源适配器
- 更完整的统一详情
- 多来源接入模板

### 路线 C：逐步走向完整 MDAMS

再往后才考虑：

- 编目模板
- OCR / AI 增强
- 更复杂的审批流
- 长期保存和迁移
- 更完整的治理体系
