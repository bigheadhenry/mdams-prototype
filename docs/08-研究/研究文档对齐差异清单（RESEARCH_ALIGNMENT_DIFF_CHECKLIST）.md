# 研究文档对齐差异清单

## 用途

本文档记录在本轮对齐之后，研究线中仍然存在、但尚未完全处理完的差异与待补项。

它不是“本轮改了什么”的流水账，而是后续继续收口研究表达时应优先检查的清单。

## 本轮已处理的核心漂移

- 已把项目从“以二维主链路为主的早期技术原型”更新为“包含二维、图像记录、统一平台、三维资源、申请交付与权限控制的研究型原型”
- 已修正标准到实现映射中关于三维实现“几乎没有直接证据”的过时描述
- 已把概念模型从单一资产/文件框架扩展为同时容纳图像记录、三维对象/版本、统一平台与权限语义
- 已补一版可直接用于论文写作的压缩摘要
- 已把核心能力压缩为 3 条标准化演示主链路和一版论文案例摘要
- 已补 IIIF 与 BagIt 的样本级证据
- 已形成一版“文档语义 -> 实施边界”的推进清单

## 当前仍需继续补强的差异

### 1. 统一对象模型仍主要停留在研究表达层
- 当前对象关系已经能解释系统，但代码层仍以二维、三维、申请和平台适配器分别实现
- 下一步更适合补共享契约或平台级 tests，而不是直接强行统一数据库模型

### 2. PREMIS 风格事件模型仍未进入跨子系统实施边界
- 文档层已有最小事件模型，三维链路也已有 `ThreeDProductionRecord`
- 但二维、申请、访问和导出链路还没有统一的事件边界或共享词表入口

### 3. 二维 profile 规则来源仍有分叉
- `metadata_layers.py`、`image_record_validation.py` 和 `reference_import.py` 都在表达 profile 完整性
- 但目前还没有单一规则来源或统一 contract tests

### 4. 三维 profile 仍偏向字段词典，而不是工作流级契约
- 三维 metadata/profile 文档和字典都已建立
- 但仍缺“最小可展示三维对象”级别的 detail / platform contract tests

### 5. IIIF 与 BagIt 还需要更强的输出层契约
- 当前已有 profile 说明和样本级证据
- 但仍需进一步用 tests 固化字段、权限、文件选择与失败模式边界

### 6. AI 辅助层的研究定位仍需保持克制
- 当前已有真实 AI 控制面板与日志链路
- 但论文表达中仍需避免让 AI 功能喧宾夺主，掩盖系统真正的核心贡献

## 后续优先级建议

1. 先统一二维 profile 必填规则来源，并补共享 contract tests
2. 再补 IIIF / BagIt 输出层 contract tests
3. 再明确跨子系统最小事件边界
4. 最后再考虑统一平台共享契约或更强三维工作流契约
