# MDAMS 概念模型（草案）

## 目的

本文档用于定义 MDAMS Prototype 当前在产品/架构层面的概念模型。

它的作用不是替代数据库 schema，而是为后续实现、文档整理与论文写作提供一个更稳定的概念层结构。

截至 **2026-04-08**，它需要同时解释二维影像、图像记录、统一平台、三维资源、申请交付与权限控制为何属于同一原型框架。

## 核心立场

当前原型应把 **数字资产（digital asset）** 视为系统中的主要管理对象。

系统不应以孤立文件为中心，也不应一开始就扩张为包罗万象的博物馆藏品管理体系；它应围绕一个可管理的数字资产对象，支撑其采集、校验、处理、访问、导出以及业务协作。

## 核心实体

### 1. 资产（Asset）
系统中的核心管理对象。

资产代表一个在逻辑上相对完整的数字对象，或一个可被系统采集、处理、访问与导出的数字资源单元。

典型职责：
- 在系统内具有身份标识；
- 关联一个或多个文件；
- 关联描述性与技术性元数据；
- 具有处理状态与可见性范围；
- 关联访问表示与导出表示。

### 2. 文件对象（File Object）
与资产相关联的物理文件对象。

可能子类型：
- 原始文件；
- 规范化文件；
- 衍生/访问文件；
- 辅助文件；
- 三维资源包中的组成文件。

典型职责：
- 存储路径/引用；
- 格式与大小；
- checksum/fixity 值；
- 与处理阶段或展示角色的关系。

### 3. 元数据记录（Metadata Record）
与资产及其文件相关联的元数据。

可能类别：
- 描述性元数据；
- 技术性元数据；
- 结构性元数据；
- 管理性元数据。

当前原型不必在所有地方一次性实现完整标准，但应清楚区分不同元数据角色。

### 4. 图像记录（Image Record）
一种与影像文件分离协作的工作流记录对象。

它不是资产本体，而是：
- 用于录入、提交、退回、指派和绑定的协作记录；
- 体现“元数据记录先行、媒体文件后绑定”的角色拆分工作流；
- 连接录入人员、摄影上传人员与后续资产生成/绑定链路的中间对象。

### 5. 三维对象与版本（3D Resource Object / Version）
三维资源并不只是单个文件，而更适合被理解为：
- 一个对象级聚合实体；
- 其下包含多个版本记录；
- 每个版本又可包含多个组成文件；
- 并可能关联 Web 展示状态与 viewer 契约。

这说明三维子系统虽然仍属原型，但已经表现出“对象 + 版本 + 文件包”的结构。

### 6. 过程事件（Process Event）
资产生命周期中被记录或可被 formalize 的事件。

例如：
- 开始采集；
- 文件分析完成；
- checksum 生成；
- 衍生文件生成；
- manifest 创建；
- 绑定/替换；
- 导出包生成；
- 交付输出；
- AI 辅助动作日志。

这个实体对系统可观测性与研究表达都非常重要。

### 7. 访问表示（Access Representation）
面向浏览或外部访问的表示层对象。

例如：
- IIIF Manifest；
- 预览图；
- viewer-ready 图像访问副本；
- 三维 viewer-ready 展示入口。

它应与原始保存导向文件区分开。

### 8. 导出包（Export Package）
面向打包、转移或交付的表示对象。

例如：
- BagIt ZIP；
- 利用申请后的交付包；
- 未来可能的结构化输出包。

它不是资产本体，而是资产在某种导出语境下的封装结果。

### 9. 平台来源与统一资源视图（Source Adapter / Unified Resource View）
统一平台层并不替代各来源子系统，而是：
- 通过来源适配器聚合不同资源系统；
- 为目录和详情提供统一视图；
- 在系统层面表达“多来源数字资源底座”的平台语义。

### 10. 权限与责任范围（Authorization / Scope）
权限不是外部附加功能，而是当前原型中的结构性语义之一。

它至少包括：
- 角色权限；
- 菜单和动作可见性；
- 责任范围过滤；
- 面向申请、查看、管理与导出的访问边界。

## 实体关系（工作解释）

当前可以用如下方式理解主要关系：

- 一个 **资产** 可以关联多个 **文件对象**；
- 一个 **资产** 可以关联一组 **元数据记录**；
- 一个 **资产** 在生命周期中会产生多个 **过程事件**；
- 一个 **资产** 可以生成一个或多个 **访问表示**；
- 一个 **资产** 可以导出为一个 **导出包**；
- 一个 **图像记录** 可以在协作流程中指向一个待绑定或已绑定的 **资产**；
- 一个 **三维对象** 可以聚合多个 **版本**，每个版本关联多个 **文件对象** 与一个可能的 **viewer 契约**；
- 一个 **统一资源视图** 可以聚合来自二维、三维等不同来源的资源摘要；
- **权限与责任范围** 影响资产、统一视图、申请流程和部分管理动作的可见性与可执行性；
- **访问表示** 与 **导出包** 都不是资产本体，而是资产的不同外部表达。

## 为什么采用这种模型

这种概念模型有几个优势：

1. 它比“文件上传系统”更强，因为它能解释元数据、过程、访问、导出与协作记录之间的关系；
2. 它比“完整藏品管理系统”更聚焦，更符合当前原型范围；
3. 它能容纳二维、图像记录、统一平台与三维资源，而不必把它们写成互相孤立的 demo；
4. 它有利于与 IIIF、BagIt、OAIS、PREMIS、NISO Z39.87、CS3DP 等标准/框架建立更清晰的对应；
5. 它能直接服务后续论文写作中的对象建模、系统说明和评估讨论。

## 当前边界

当前概念模型仍是工作草案，尚未完全细化：
- `Asset` 与 `ImageRecord` 的正式关系仍需进一步澄清；
- 三维对象、版本和文件包是否需要单独抽象成更强实体体系仍待决定；
- 过程事件尚无正式事件词表；
- 访问表示与导出包尚未统一 formalize；
- 藏品对象、申请对象与资源对象之间的关系仍需进一步明确。

## 当前判断

从产品与研究两个角度看，**资产（Asset）作为核心对象** 仍是当前最稳妥的方向。

但截至 **2026-04-08**，这个判断需要进一步补充为：

> MDAMS 是一个以数字资产为核心，同时通过图像记录协作、统一平台聚合、三维对象版本管理、访问表示与导出表示，把多个数字资源工作流组织到同一原型框架中的博物馆/展陈数字资源管理原型。
