# 当前项目事实（供研究写作用）

## 目的

本文档用于记录截至 **2026-04-08** 仍能被当前仓库、项目文档和工作日志直接支持的项目事实。

它服务于后续研究写作、论文表达和项目对外说明，因此优先记录：
- 已实现并可验证的事实；
- 当前最强的演示链路；
- 当前明确存在的边界与缺口；
- 能直接转化为论文论述的证据点。

## 当前项目定位

MDAMS Prototype 当前最适合被理解为一个**面向博物馆/展陈数字资源场景的研究型技术原型**，而不是一个完整生产级 DAMS。

与早期“二维影像上传 PoC”的说法相比，截至 **2026-04-08** 的更准确表述是：

> 一个已经具备二维影像、图像记录协作、统一平台聚合、三维资源、申请交付、认证权限与基础测试体系的数字资源管理原型底座。

它当前最强的研究价值，不在于功能面无限扩张，而在于已经形成：
- 一条可解释、可演示、可重复验证的核心工作流；
- 一个以数字资产为中心的原型倾向；
- 一组与 IIIF、BagIt、OAIS、PREMIS、NISO Z39.87 等标准/框架的分层关系；
- 以及一个具有继续扩展能力的真实工程底座。

## 当前已实现能力

根据 `README.md`、`docs/01-总览/PROJECT_STATUS.md`、`docs/01-总览/WORK_LOG.md` 以及当前代码结构，当前已实现范围至少包括以下板块。

### 1. 二维影像子系统
- 资产上传、列表、详情、删除；
- 预览图生成与列表缩略图展示；
- 动态 IIIF Manifest 生成；
- Mirador 预览；
- 原文件下载与 BagIt 下载；
- 大图像访问副本策略与 IIIF access 流程；
- PSB / PSD / TIFF 等图像导向处理链路。

### 2. 图像记录工作流
- 基于 `ImageRecord` 的角色拆分工作流；
- 录入人员创建、编辑、提交、退回记录；
- 摄影上传人员待上传池；
- 临时上传分析、显式绑定与替换；
- 绑定后校验、重复检测与状态推进。

### 3. 统一平台层
- 平台来源适配器与注册表；
- 统一资源目录；
- 统一资源详情；
- 按状态、类型、profile、预览能力筛选；
- 至少已聚合二维与三维两类来源。

### 4. 三维资源子系统
- 三维对象与版本管理；
- 多文件资源包上传与管理；
- 模型 / 点云 / 倾斜摄影等角色区分；
- viewer 契约与 Web 预览；
- 对象级聚合视图；
- 平台接入与统一浏览。

### 5. 利用申请与交付
- 申请车；
- 申请单创建；
- 审批通过 / 拒绝；
- 交付包导出。

### 6. 认证、权限与责任范围控制
- 登录、会话、上下文接口；
- 内置测试用户与角色播种；
- 后端基于角色权限做访问判断；
- 前端菜单按权限裁剪；
- `collection_owner` 基于责任范围控制资源可见性；
- `owner_only` 访问边界已进入后端判断与前端回归测试。

### 7. 测试与工程支撑
- 后端 `pytest` 测试；
- 前端 `Playwright` 回归测试；
- Docker Compose 部署结构；
- PostgreSQL 测试库基线；
- Redis、Celery、Cantaloupe 等服务协同；
- 参考资源导入与日志记录机制。

### 8. AI 辅助交互层
- Mirador AI 控制面板；
- 自然语言控制缩放、平移、适配、重置；
- 候选图搜索与确认；
- 比较模式与动作日志；
- OpenAI 兼容模型服务接入。

当前最稳妥的研究表述是：AI 功能已经形成真实实现证据，但其定位更适合作为**辅助交互层与实验性增强层**，而不是系统主贡献。

## 当前最强可演示工作流

截至 **2026-04-08**，当前最有代表性的演示链路已经不止一条，而是至少包括以下三类：

### A. 二维数字资产主链路
1. 上传数字文件或 ingest 进入系统；
2. 生成 fixity、提取技术元数据；
3. 必要时触发异步处理与访问副本生成；
4. 生成面向 IIIF 的访问表示；
5. 在 Mirador 中进行预览；
6. 以 BagIt 导向方式导出结果。

### B. 图像记录协作链路
1. 录入人员创建并提交 `ImageRecord`；
2. 摄影上传人员在待上传池中查看记录；
3. 进行文件上传、分析、绑定或替换；
4. 系统执行绑定后校验与状态推进；
5. 进入访问、管理和后续交付链路。

### C. 三维对象链路
1. 创建三维对象并上传多文件资源包；
2. 管理版本、Web 展示状态与 viewer 契约；
3. 在三维管理页和统一平台中查看聚合对象；
4. 对可展示对象进行 Web 预览和统一浏览。

## 当前架构事实

根据当前仓库，系统主要架构包括：

- **前端**：React + Vite + TypeScript + Ant Design
- **二维查看器**：Mirador
- **三维查看器**：`@google/model-viewer`
- **后端**：FastAPI + SQLAlchemy
- **任务处理**：Celery + Redis
- **数据库**：PostgreSQL
- **IIIF 图像服务**：Cantaloupe
- **部署方式**：Docker Compose

## 当前对象模型事实

截至当前，最可靠的研究事实不是“对象模型已经完全澄清”，而是：

1. 系统明显呈现出以 **`Asset` / 数字资产** 为中心的组织倾向；
2. 二维、图像记录、统一平台、申请交付和三维资源并非互不相关的孤立 demo，而是逐步围绕统一数字资源底座收束；
3. 图像记录工作流显示出“元数据记录”和“影像文件”分离协作的趋势；
4. 三维子系统显示出“数字对象 + 版本 + 文件包 + viewer 契约”的结构；
5. 统一平台显示出“来源适配器 + 聚合目录 + 统一详情”的平台层趋势。

这意味着后续研究不应再把项目简单描述为“资产上传系统”，而应进一步解释这些对象和关系如何被同一原型框架容纳。

## 当前最值得强调的研究事实

从研究表达角度，当前最值得抓住的事实包括：

1. 该项目不是抽象设想，而是有真实代码、真实架构、真实部署路径、真实测试与真实工作日志的系统；
2. 它已经不再只是二维影像上传 PoC，而是包含二维、图像记录、统一平台、三维、申请交付、权限控制等多模块原型；
3. 它最强的价值仍然不在“模块越多越好”，而在形成可演示、可解释、可验证的主链路；
4. 它与 IIIF、BagIt、OAIS、PREMIS、NISO Z39.87 等标准/框架的关系是分层的，而不是笼统的“全部遵循”；
5. 它更适合被解释为一个**具有保存意识的数字资产管理原型**，而不是普通上传下载平台；
6. 它已经具备继续发展为研究案例和持续工程底座的条件。

## 当前主要缺口

虽然当前功能和文档都已明显超出早期 PoC，但截至 **2026-04-08** 仍存在以下关键缺口：

- 核心对象模型尚未完全收口；
- `Asset`、`ImageRecord`、藏品对象、三维对象/版本、访问表示、导出包之间的关系仍需进一步明确；
- 标准到实现映射仍需更细粒度 formalization；
- PREMIS 风格事件模型仍未显式建立；
- 图像技术元数据与三维元数据 profile 仍待系统化；
- 权限、治理、审计、迁移与长期保存层仍未形成完整体系；
- 研究表达仍需持续与实现事实对齐，避免文档漂移。

## 当前工作判断

基于现有事实，MDAMS 的下一阶段研究表达不应停留在“系统做了哪些功能”，而应进一步回答：
- 为什么围绕数字资产组织系统；
- 为什么要同时容纳二维、图像记录、统一平台与三维资源；
- 为什么采用选择性标准对齐；
- 如何把当前实现解释为一个具有保存意识、但边界明确的研究型原型案例。
