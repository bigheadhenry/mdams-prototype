# 标准到实现映射

## 目的

本文档把 MDAMS Prototype 当前实现与已识别的标准、参考模型和社区框架进行直接对应。

截至 **2026-04-08**，它优先回答四个问题：
1. 这个标准/框架在项目里扮演什么角色？
2. 当前实现中已经具备了什么？
3. 还缺什么，或者哪些只是隐含存在？
4. 如果要增强对齐，下一步最现实的动作是什么？

---

## 映射级别说明

- **直接对齐**：系统里已经存在明确实现特征、接口、输出或可验证工作流
- **部分对齐**：实现行为与标准有对应，但还未显式 formalize 为稳定模型或 profile
- **概念对齐**：主要作为解释框架使用，而不是直接实现标准
- **未来导向**：当前主要用于指导下一阶段扩展

---

## 1. IIIF

### 角色
访问/展示互操作标准。

### 当前对齐级别
**直接对齐**

### 当前实现中已经存在
- 动态 IIIF Manifest 生成；
- 面向 IIIF 的访问 URL 结构；
- 基于 Cantaloupe 的图像服务集成；
- Mirador 查看器集成；
- 图像访问副本与原始保存导向文件的区分；
- 围绕资产与访问表示组织的实际浏览链路。

### 还缺什么
- 当前实际支持的 IIIF Manifest profile 仍需稳定说明；
- ranges、annotations、多部件对象等更复杂结构的支持情况仍不清晰；
- 尚缺明确的 Manifest 输出验证流程。

### 实际解释
IIIF 仍然是当前 MDAMS 中最强、最直接、最可展示的标准实现层。

### 下一步建议
- 文档化当前 Manifest 输出 profile；
- 标出已支持、可选支持、未支持字段；
- 增加基础 Manifest 验证说明。

---

## 2. BagIt

### 角色
面向保存导向传输与导出的打包标准。

### 当前对齐级别
**直接对齐**

### 当前实现中已经存在
- BagIt ZIP 导出；
- 导出逻辑围绕数字资产或交付结果展开，而非简单附件下载；
- 与 fixity 导向和导出/交付链路相连；
- 能作为“保存导向输出表示”的真实系统证据。

### 还缺什么
- bag 结构说明仍不完整；
- manifest 算法、校验机制与关键文件说明仍需系统记录；
- 尚未形成 bag profile 级别的稳定定义。

### 实际解释
BagIt 是已经落实到系统功能中的直接对齐点，但当前不应把它夸大为完整保存打包体系。

### 下一步建议
- 检查真实导出包结构；
- 记录 BagIt 关键文件与校验规则；
- 增加 bag profile 级说明。

---

## 3. OAIS

### 角色
概念性的数字保存参考模型。

### 当前对齐级别
**概念对齐**

### 当前实现中已经存在
- ingest 导向的过程语言与接口；
- 生命周期意识；
- 原始文件、访问表示、导出表示之间的区分；
- 保存导向的打包与输出思路；
- 图像记录和交付流程中可见的阶段性工作流。

### 还缺什么
- 没有正式 OAIS 信息模型映射；
- 没有把 SIP / AIP / DIP 显式建模为一等实体；
- 未覆盖 OAIS 全部职能域；
- 还未形成正式的 OAIS 范围图。

### 实际解释
OAIS 当前更适合作为研究解释框架，而不是系统实现标准。

### 下一步建议
- 画一张轻量 OAIS 范围对照图；
- 明确哪些 OAIS 功能在当前原型边界之外。

---

## 4. PREMIS

### 角色
以对象、事件、代理、权利为核心的保存元数据框架。

### 当前对齐级别
**部分对齐**

### 当前实现中已经存在
- 以资产为中心的记录，可视作 proto-object；
- fixity 生成/校验；
- 元数据提取；
- 异步处理与转换；
- IIIF 访问副本生成；
- 导出/交付；
- 时间戳、状态字段与部分操作日志；
- AI 辅助链路中的动作日志；
- 一定程度上可识别的系统、用户与处理过程角色。

### 还缺什么
- 尚未把 Object / Event / Agent / Rights 正式建模；
- 没有稳定事件词表；
- 操作历史更多是工程性存在，而不是正式保存元数据；
- rights 维度仍较弱；
- 三维生产链路和图像记录链路还未统一纳入同一事件模型。

### 实际解释
当前 MDAMS 已具有多种 **proto-PREMIS 行为**，尤其体现在对象状态、处理动作和日志痕迹层面，但尚未达到正式 PREMIS 实现。

### 下一步建议
- 建立最小 PREMIS 风格事件模型；
- 为 ingest、checksum、metadata extraction、conversion、iiif derivative generation、manifest generation、export、delivery 建立事件词表；
- 明确最小 agent / rights / object 信息集合。

---

## 5. NISO Z39.87

### 角色
数字静态图像技术元数据标准。

### 当前对齐级别
**部分对齐**

### 当前实现中已经存在
- 图像元数据提取；
- 图像导向工作流；
- PSB / PSD / TIFF 等大图像处理；
- 与 IIIF 图像访问链相关的图像特征处理；
- 图像记录提交校验与绑定后校验中的技术元数据语义。

### 还缺什么
- 尚无显式字段 crosswalk；
- 当前提取字段覆盖到 Z39.87 哪些层级仍需盘点；
- 尚未形成稳定的图像技术元数据 profile。

### 实际解释
当前系统已经是“图像技术元数据感知”的，但组织方式仍更偏实现驱动，而不是标准 profile 驱动。

### 下一步建议
- 盘点当前图像元数据提取字段；
- 与 Z39.87 做最小 crosswalk；
- 形成 still image 的 minimum viable profile。

---

## 6. CS3DP（3D Data Creation to Curation / Community Standards for 3D Data Preservation）

### 角色
围绕 3D 数据生命周期、元数据、保存、权利与访问的社区指导框架。

### 当前对齐级别
**未来导向，带有限实现参照意义**

### 当前实现中已经存在
- 三维对象与版本管理；
- 多文件资源包结构；
- 模型 / 点云 / 倾斜摄影等角色区分；
- viewer 契约与 Web 展示状态；
- 生产链路记录与保存层/展示层分离倾向；
- 统一平台中的三维来源聚合。

### 还缺什么
- 尚未形成明确的 3D 元数据 profile；
- 尚未建立稳定的 3D 保存工作流 formalization；
- 尚未把 3D 对象、事件、权利与长期保存要求映射到统一研究模型；
- 当前实现距离社区标准层面的系统化对齐仍有明显距离。

### 实际解释
截至 **2026-04-08**，CS3DP 已不应被描述为“当前几乎没有直接实现证据”的纯未来材料。更准确的说法是：

> MDAMS 已经出现真实的三维对象、版本、文件包和 Web 展示实现，因此 CS3DP 对当前系统已具有初步参照意义；但这种关系仍主要是 roadmap 级和解释级，而不是正式标准对齐。

### 下一步建议
- 把当前三维对象、版本、文件包和展示状态整理成最小 3D 对象模型；
- 为三维元数据建立 minimum viable profile；
- 再决定是否进一步引入更正式的 CS3DP 映射。

---

## 跨标准总体判断

截至 **2026-04-08**，MDAMS 与不同标准/框架的关系更准确的分层解释是：

- **IIIF** = 最强直接实现层；
- **BagIt** = 直接导出/打包层；
- **OAIS** = 概念解释层；
- **PREMIS** = 保存元数据的部分对齐层；
- **NISO Z39.87** = 图像技术元数据的部分对齐层；
- **CS3DP** = 三维扩展的未来导向参考层，但已具有限实现参照意义。

这种解释优于笼统说“系统遵循多个标准”，因为它清楚区分了：
- 已经实现的标准能力；
- 借鉴的概念框架；
- 尚待 formalization 的潜力；
- 未来扩展方向。

---

## 研究意义

这份映射支撑一个关键研究判断：

> MDAMS Prototype 的价值在于，它把若干关键互操作/导出标准的直接实现，与保存和元数据框架的选择性吸收结合起来，从而形成一个既有工程可演示性、又具研究解释力的数字资产管理原型。
