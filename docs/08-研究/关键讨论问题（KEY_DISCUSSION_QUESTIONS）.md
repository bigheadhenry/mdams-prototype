# 关键讨论问题（KEY_DISCUSSION_QUESTIONS）

## 用途

本文档用于把当前 MDAMS Prototype 后续最值得正式讨论的 5 个问题固定下来，避免讨论再次退化为“还要不要继续加功能”。

这 5 个问题都来自当前项目已经形成的原则、研究边界和实施计划。它们不是开放式头脑风暴，而是后续会直接影响：
- 代码结构如何继续推进；
- 测试与验证如何补强；
- 论文如何诚实而有力地表述；
- 原型边界如何保持清楚。

## 讨论前提

讨论这些问题时，应保持以下前提不变：

1. MDAMS 当前仍是研究型原型，而不是生产级系统；
2. 后续重点应是固化语义与验证边界，而不是横向扩功能；
3. 所有讨论都应同时服务实现线与论文线；
4. 如果一个方案会明显提高复杂度，但不能增加研究解释力或验证证据，应谨慎采用。

## 问题 1：跨子系统最小事件边界到底怎么定义？

### 当前事实

- 三维链路已有 `ThreeDProductionRecord`，已经接近正式事件对象；
- 二维、申请、IIIF、BagIt 等链路主要还是状态字段、detail 派生说明和动作输出；
- 研究线中已经形成 `PREMIS事件映射（PREMIS_EVENT_MAPPING）.md`，但代码层尚未形成跨子系统一致边界。

### 为什么必须讨论

如果不先定义“什么算事件”，后续关于治理、保存、审计、责任追踪的讨论都会变得不稳定。

更具体地说，当前系统里同时存在：
- 状态；
- 处理步骤；
- 输出动作；
- 局部事件对象。

如果这些层级不先分清，论文很容易过度表述，实现也容易无序扩张。

### 可选方案

1. 先定义最小事件分类，并优先落到 detail / test 层
2. 直接设计跨子系统统一事件表
3. 继续保持当前分散表达

### 推荐立场

优先选择 **方案 1**。

先把不同链路的最小事件边界、事件词表和 detail/test 表达收口，再决定是否真的需要统一事件持久化。

### 对实现的影响

- 可以先低成本统一语义；
- 不会立刻触发数据库级重构；
- 为后续治理层或保存层预留更清晰入口。

### 对论文的影响

- 可以诚实表述“已有最小事件边界”；
- 避免把当前原型误写成“已有完整事件审计体系”。

### 讨论时要回答的具体问题

1. 哪些动作必须作为事件保留？
2. 哪些只需要作为状态即可？
3. 哪些只应作为输出层派生说明？
4. detail 层是否应显式暴露统一事件摘要？

## 问题 2：统一对象模型要统一到什么程度？

### 当前事实

- 研究文档已经能把 `Asset`、`ImageRecord`、三维对象/版本、访问表示、导出表示和权限范围解释为一个整体；
- 代码层仍以二维、三维、申请和平台适配器分治；
- 当前平台聚合已经能提供“统一目录/详情”，但这不等于底层对象已统一。

### 为什么必须讨论

后面如果不澄清“统一”的层级，团队很容易在两条风险路径中摇摆：
- 要么过早重构底层模型；
- 要么在研究表达里把“平台聚合”误写成“对象模型已统一”。

### 可选方案

1. 保持分子系统模型，只补共享契约
2. 加一层统一领域抽象
3. 继续推进到底层数据库模型统一

### 推荐立场

优先选择 **方案 1**，必要时局部采用 **方案 2**。

当前不建议为了形式上的统一去重构数据库结构，更合理的做法是先固化共享字段、共享动作和平台级契约。

### 对实现的影响

- 保持现有模块稳定；
- 更适合通过平台层和 detail 层继续固化一致性；
- 可避免高风险迁移。

### 对论文的影响

- 可以更准确地表述为“语义统一、存储分治”；
- 有利于强调原型的可扩展性而非假装彻底统一。

### 讨论时要回答的具体问题

1. “统一对象”是研究层概念，还是代码层接口？
2. 平台层最小共享字段集合是什么？
3. 未来扩更多来源时，统一契约如何继承？

## 问题 3：最小 profile 和字段约束应该固化到哪一层？

### 当前事实

- 二维 profile 最小必填规则已开始收口到共享服务层；
- 三维 profile 目前更偏字段词典、分层 metadata 和说明文档；
- 研究线已经把 metadata/profile 说成系统重要边界，但不同约束还没有完全进入同一层级。

### 为什么必须讨论

如果不先定义“哪些字段是硬约束，哪些字段是软约束”，后面会出现两类问题：
- 服务层校验越来越重，原型被过度 formalize；
- 或者研究文档写得很完整，但系统行为并没有真正体现这些约束。

### 可选方案

1. 把“最小可运行/可提交”字段固化到服务层，其余落到 detail/test 层
2. 大部分字段只做展示，不做强约束
3. 全量 schema 化并强校验

### 推荐立场

优先选择 **方案 1**。

只把最小闭环所必需的字段做硬约束，其余先通过 detail 输出和 contract tests 固化。

### 对实现的影响

- 主链路更稳定；
- 不会因为追求完整 schema 而拖慢开发；
- 更适合研究原型阶段的节奏。

### 对论文的影响

- 有利于区分“最低约束”和“扩展语义”；
- 能更准确说明项目的标准对齐深度。

### 讨论时要回答的具体问题

1. 二维和三维各自的最小必填字段集合是什么？
2. 哪些字段必须阻断提交？
3. 哪些字段只需要在导入完整性或 detail 层提示？
4. contract tests 要覆盖到什么程度？

## 问题 4：IIIF / BagIt 这类输出层后面保持派生对象，还是进一步结构化？

### 当前事实

- IIIF Manifest 和 BagIt 已有明确实现、样本级证据和 contract tests；
- 当前架构把它们处理为访问表示和导出表示，而非独立持久化对象；
- 研究表达已经开始明确“输出层边界”。

### 为什么必须讨论

这是一个很典型的系统边界问题。

如果讨论不清楚，后面很容易在两种误区之间摇摆：
- 要么把输出层过度对象化，增加复杂度；
- 要么继续只靠文档说明，缺少足够稳的系统边界。

### 可选方案

1. 继续保持派生表示，只强化输出契约和失败模式
2. 上升为第一类持久化对象
3. 采用混合模式，只对象化部分输出

### 推荐立场

优先选择 **方案 1**。

当前实现和研究表达都更适合把 IIIF / BagIt 视为“可重复生成的派生输出表示”。

### 对实现的影响

- 与现有路由和 detail 模型一致；
- 可以继续通过 tests 固化边界，而不引入额外对象复杂度；
- 更利于后续按需求局部增强。

### 对论文的影响

- 可以明确区分“资源对象”和“访问/导出表示”；
- 有助于解释系统为何具有保存意识但仍保持原型边界。

### 讨论时要回答的具体问题

1. 哪些输出字段属于必须稳定的契约？
2. 哪些失败模式必须显式暴露？
3. 输出层是否需要对应事件摘要？
4. 何时才值得把输出表示升级为独立对象？

## 问题 5：下一阶段的完成标准到底是什么？

### 当前事实

- 项目现在既有研究文档，也有多条真实业务链路，还有测试；
- 但如果没有统一完成标准，后续工作仍可能重新回到分散推进；
- 当前已经形成 `memory/`、`tasks/` 和实施边界清单，说明项目正在向更稳定的推进方式过渡。

### 为什么必须讨论

后续如果不先定义“什么叫完成一轮工作”，团队就很容易继续：
- 补更多说明；
- 加更多零散功能；
- 跑部分验证；
- 但没有形成一轮真正闭环。

### 可选方案

1. 以“实现边界 + 测试证据 + 写作摘要”三件套作为完成标准
2. 以功能新增作为完成标准
3. 以论文章节推进作为完成标准

### 推荐立场

优先选择 **方案 1**。

这最符合研究型原型项目的双轨目标，也最符合当前仓库已经建立的工作方式。

### 对实现的影响

- 每轮都能留下稳定实现结果；
- 每轮都能留下验证证据；
- 每轮都能自然接到下一任务。

### 对论文的影响

- 每轮都能留下可复用材料；
- 不会再依赖聊天历史或临时回忆整理论证。

### 讨论时要回答的具体问题

1. 每轮最少需要哪些测试或验证结果？
2. 需要同步更新哪些 `memory/` 或 `docs/` 文件？
3. 哪类工作必须留下论文可复用摘要？
4. 何时可以判定某个边界“已经稳定”？

## 推荐讨论顺序

建议按以下顺序讨论：

1. 跨子系统最小事件边界
2. 统一对象模型的落点
3. 最小 profile 约束层级
4. IIIF / BagIt 输出层的长期定位
5. 下一阶段完成标准

这个顺序的原因很简单：
- 事件边界会影响对象模型表达；
- 对象模型会影响 profile 和输出层定位；
- 前四项明确后，完成标准才能真正落地。

## 当前建议结论

如果只保留一句面向下一阶段的工作原则，可以写成：

> MDAMS Prototype 后续最重要的工作，不是继续横向扩展功能，而是围绕事件边界、对象模型、最小 profile、输出层定位和完成标准这 5 个问题，逐步把研究语义推进到可验证、可复用、可写作的实施边界。
