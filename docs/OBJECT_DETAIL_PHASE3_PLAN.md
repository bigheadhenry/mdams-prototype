# MDAMS 资源对象详情第三轮实现计划：处理轨迹与生命周期可视化

## 1. 本轮目标

在第二轮已经完成“对象结构 / 访问路径 / 输出动作”整理的基础上，本轮目标是为当前对象详情页补上一个最小可用的**处理轨迹 / 生命周期记录**区块，使对象页不再只是静态结构展示，而开始具备“对象经历了哪些关键处理步骤”的表达能力。

本轮目标不是建设完整事件系统，而是先完成：

> 让单件二维文物图片资源对象在详情页中呈现一个可读、可解释、可继续扩展的最小生命周期轨迹。

---

## 2. 本轮问题背景

当前对象详情页已经能回答：

- 它是什么对象；
- 它目前有哪些文件角色；
- 它有哪些访问路径；
- 它有哪些输出动作。

但它还不太能回答：

- 这个对象经历了哪些处理步骤；
- 当前“已就绪 / 可预览”是如何形成的；
- 是否已经完成 fixity、元数据提取、访问副本形成、输出能力准备。

因此，本轮需要补一个最小的“过程层表达”。

---

## 3. 本轮实现原则

### 3.1 不引入完整事件存储系统

本轮**不**做以下内容：

- 不新增事件表；
- 不引入完整 PREMIS 事件持久化；
- 不建设后台审计日志系统；
- 不实现跨对象历史检索。

### 3.2 优先通过详情接口聚合推导最小轨迹

本轮优先在 `GET /assets/{id}` 中基于现有字段、元数据和当前对象状态，推导一个轻量的 `process_timeline` 结构。

### 3.3 面向当前阶段对象闭环表达

本轮只面向当前阶段闭环：

> 上传 / 入库 → 基础处理 → 可访问 → 可输出

因此轨迹节点应围绕这个闭环组织，而不是过早做成通用复杂流程引擎。

---

## 4. 后端改动方案

在 `GET /assets/{asset_id}` 返回结构中新增：

## 4.1 `process_timeline`

建议返回一个列表，每项至少包含：

- `step`
- `label`
- `status`
- `status_label`
- `description`

### 初始最小节点建议

1. `object_created`
   - 对象记录已建立
2. `ingest_completed`
   - 文件已上传或已完成 SIP 入库
3. `fixity_recorded`
   - 已记录 fixity / SHA256（若存在）
4. `metadata_extracted`
   - 已形成基础技术元数据（宽高等）
5. `access_copy_ready`
   - 当前访问主文件已准备好
6. `preview_ready`
   - 已可通过 IIIF / Mirador 访问
7. `output_ready`
   - 当前文件下载与 BagIt 输出入口已具备

### 状态规则

状态先只使用轻量三值：

- `done`
- `pending`
- `error`

若对象整体状态为 `error`，则相关节点可标注为 `error` 或保留 `pending`。

---

## 5. 前端改动方案

在 `AssetDetail.tsx` 中新增独立区块：

## 5.1 “处理轨迹”

展示方式采用最稳定的简单列表，不引入复杂时间轴依赖。

每一步展示：

- 步骤名称
- 状态标签
- 简短说明

状态颜色建议：

- `done` = 绿色
- `pending` = 默认 / 蓝色
- `error` = 红色

## 5.2 页面位置

建议放在：

- 对象摘要之后，
- 对象结构之前。

这样页面阅读顺序就变成：

1. 它是什么
2. 它走到了哪一步
3. 它内部是什么结构
4. 它如何被访问
5. 它如何被输出

---

## 6. 验证目标

本轮至少验证：

1. `GET /assets/{id}` 已返回 `process_timeline`；
2. 不同对象（普通上传 / SIP 入库）都能看到轨迹区块；
3. `ready` 对象显示多数步骤为已完成；
4. 无 SHA256 或无原始转换信息的对象，也能稳定退化展示；
5. 不破坏现有详情页、下载、Manifest、BagIt 功能。

---

## 7. 本轮成果预期

本轮完成后，当前 MDAMS 资源对象详情页将从：

> 静态对象结构表达

进一步提升为：

> 同时具备“对象结构 + 最小生命周期轨迹”表达能力的资源对象页

这将为后续可能继续推进的方向打基础：

- 更细的处理状态与失败节点表达；
- PREMIS 事件模型的未来接入；
- 多资源类型对象的差异化生命周期展示。
