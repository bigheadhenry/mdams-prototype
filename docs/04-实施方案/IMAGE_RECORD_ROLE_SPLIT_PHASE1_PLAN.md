# IMAGE_RECORD_ROLE_SPLIT_PHASE1_PLAN

## Purpose

This document archives the agreed conclusions for Problem 1: separating image metadata entry from image upload/matching. It is intended to serve as the implementation baseline for the next development thread.

## Problem Statement

The current 2D image ingest flow is file-centric. It does not reflect the real operational split between:

- metadata entry staff, who create and maintain image information records
- photographers, who upload image files and match them to pre-created records

The system therefore needs a role-split workflow in which metadata and files are managed as two linked but distinct objects.

## Final Agreed Conclusions

### 1. New Roles

Two new roles must be added to the existing permission system:

- `image_metadata_entry`
- `image_photographer_upload`

### 2. Workflow Split

The workflow is split into two stages:

1. Metadata entry staff create and submit an image record.
2. Photographers upload a file and match it to that record.

### 3. New Business Object

A new object `ImageRecord` must be introduced.

`ImageRecord` represents one image metadata record for one image object. It is not a multi-image container.

### 4. Relationship Between Record and Asset

The relationship between `ImageRecord` and `Asset` is strictly `1:1`:

- one `ImageRecord` can bind to at most one `Asset`
- one `Asset` can belong to only one `ImageRecord`

This should be enforced in business logic first and ideally by a unique constraint on `assets.image_record_id`.

### 5. Multi-image Cultural Object Handling

A cultural object may have multiple image objects, but this is not modeled as a shared metadata container in phase 1.

For now:

- each image remains an independent `ImageRecord`
- weak association across images can be maintained through fields such as `object_number`
- no automatic grouping or shared metadata propagation is required in phase 1

### 6. Record Number

`record_no` should:

- be auto-generated by the system
- remain editable by staff when needed

### 7. Replacement Rule

If an image is already matched to an `ImageRecord`, the photographer is allowed to replace it.

Replacement rules:

- replacement is allowed
- a second active image binding is not allowed
- replacement operations must leave an audit trail

## Phase 1 Scope

Phase 1 only covers metadata-entry-side enablement and record preparation for later upload matching.

Included:

- new roles and permissions
- new `ImageRecord` model
- metadata entry workbench
- image record state flow
- photographer-visible ready-for-upload record pool foundation
- `Asset.image_record_id` preparation for 1:1 binding

Excluded:

- automatic grouping of multiple images
- shared metadata across image groups
- batch auto-matching
- advanced collaboration orchestration

## Recommended Data Model

### New Table: `image_records`

Recommended core fields:

- `id`
- `record_no`
- `title`
- `status`
- `resource_type`
- `visibility_scope`
- `collection_object_id`
- `profile_key`
- `metadata_info`
- `created_by_user_id`
- `submitted_by_user_id`
- `assigned_photographer_user_id`
- `created_at`
- `updated_at`
- `submitted_at`

### Update Existing Table: `assets`

Add:

- `image_record_id`

Recommended rule:

- `assets.image_record_id` should be unique

## Recommended Status Flow for Phase 1

Minimal states:

- `draft`
- `ready_for_upload`
- `returned`

Meaning:

- `draft`: metadata record is still being edited
- `ready_for_upload`: metadata record is complete enough for photographer matching
- `returned`: record was sent back for correction

## Recommended Permission Set

### Role: `image_metadata_entry`

Recommended permissions:

- `image.record.create`
- `image.record.view`
- `image.record.edit`
- `image.record.submit`
- `image.record.return`
- `image.record.list`

### Role: `image_photographer_upload`

Recommended permissions:

- `image.record.view`
- `image.record.view_ready_for_upload`
- `image.file.upload`
- `image.file.match`

Constraint:

- photographers should not edit core business metadata directly in phase 1

## Recommended Backend Work for Phase 1

Suggested new router:

- `backend/app/routers/image_records.py`

Suggested endpoints:

1. `POST /api/image-records`
2. `GET /api/image-records`
3. `GET /api/image-records/{id}`
4. `PATCH /api/image-records/{id}`
5. `POST /api/image-records/{id}/submit`
6. `POST /api/image-records/{id}/return`
7. `GET /api/image-records/ready-for-upload`

## Recommended Frontend Work for Phase 1

Suggested new views:

- `ImageRecordList`
- `ImageRecordForm`
- `ImageRecordDetail`

Suggested menu entry for metadata entry staff:

- `影像信息录入`

Suggested form structure:

- core section
- management section
- profile section

## Required Field Strategy

Draft save should be permissive.

Submit-to-upload should enforce a minimal required set.

Recommended common required fields:

- `title`
- `visibility_scope`
- `profile_key`
- `project_name`
- `image_category`

Recommended profile-specific required fields:

- `business_activity`: `main_location`
- `movable_artifact`: `object_name`
- `immovable_artifact`: `building_name`
- `ancient_tree`: `archive_number`

## Implementation Order

Recommended sequence:

1. database schema and model
2. role and permission updates
3. backend schemas and router
4. frontend list page
5. frontend form page
6. submit action and validation
7. detail page
8. menu and permission visibility integration

## Acceptance Criteria for Phase 1

Phase 1 is complete when:

1. metadata entry staff can log in and access the image-record entry area
2. a new `ImageRecord` can be created and saved as draft
3. layered metadata can be edited and persisted on the record
4. a record can be submitted into `ready_for_upload`
5. photographers can query the ready-for-upload record pool
6. the system is prepared for later 1:1 `ImageRecord` to `Asset` binding

## Out-of-Scope Reminder

The following are intentionally deferred:

- multi-image grouping
- shared metadata inheritance across grouped images
- batch auto-group creation
- automatic multi-file binding

These can be revisited in later phases after single-image workflow stabilization.
