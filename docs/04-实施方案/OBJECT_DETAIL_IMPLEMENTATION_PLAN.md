# MDAMS 资源对象详情页与闭环收口实现计划

## 1. 文档目标

本文档用于将当前 MDAMS Prototype 从“上传 / 列表 / 预览 / 下载的技术能力集合”进一步收束为一个更清晰的**单件二维文物图片资源对象闭环原型**。

本轮实现不追求全面重构数据模型，也不试图一次性完成完整业务系统，而是采用**最小改动、最大复用**的策略，在现有代码基础上补足：

- 资源对象详情视图；
- 更清晰的处理状态表达；
- 更明确的访问与输出入口；
- 上传后直接进入对象详情的演示路径。

---

## 2. 当前判断

当前项目已经具备以下能力：

- 上传文件并写入 `Asset` 记录；
- 提取基础图像尺寸信息；
- 通过 SIP 入库链路执行 fixity 校验与元数据提取；
- 对 PSB 执行异步 BigTIFF 转换；
- 动态生成 IIIF Manifest；
- 在前端通过 Mirador 预览；
- 生成并下载 BagIt ZIP 包。

但当前原型仍主要表现为：

> 一组技术功能点的串联。

距离“面向对象的、可解释的、可演示的原型闭环”还差一个关键层：

> **以对象详情页为中心的资源对象表达层。**

---

## 3. 本轮实现目标

本轮实现目标定义为：

> 将当前 `Asset` 在第一阶段视为“单件二维文物图片资源对象”的承载记录，并通过详情接口、详情页与更明确的状态/访问/输出入口，把主链路收束为可演示闭环。

本轮完成后，期望形成以下路径：

1. 上传图片；
2. 创建对象记录；
3. 跳转到对象详情页；
4. 在详情页中查看对象摘要、文件信息、技术元数据、处理状态；
5. 从详情页进入 IIIF / Mirador 访问；
6. 从详情页触发下载/导出动作。

---

## 4. 实施原则

### 4.1 不做大重构

本轮不做以下事项：

- 不将 `Asset` 全面重构为多表 `ResourceObject / File / Representation / MetadataRecord` 体系；
- 不大规模改造前端为完整多路由业务应用；
- 不改动当前 IIIF / Manifest / Nginx 相关链路语义；
- 不同时展开多对象类型支持。

### 4.2 优先复用现有能力

本轮优先复用：

- 现有 `Asset` 模型；
- 现有 `/assets` 列表接口；
- 现有 `/iiif/{asset_id}/manifest`；
- 现有 Mirador 预览方式；
- 现有 BagIt 下载能力。

### 4.3 先实现“对象感”，再追求模型纯度

当前阶段重点不是命名是否绝对完美，而是先让系统从用户视角成为：

> 一个资源对象进入系统后可被查看、理解、访问和输出。

因此，第一阶段可以接受：

> `Asset` = 当前阶段的“单件二维文物图片资源对象”承载记录。

---

## 5. 后端改动方案

## 5.1 Asset 模型轻量增强

建议在 `backend/app/models.py` 中为 `Asset` 增加以下字段：

- `resource_type`：默认值 `image_2d_cultural_object`
- `process_message`：处理提示或错误说明，可为空

目的：

- 明确当前对象类型；
- 让前端可直接显示更友好的状态说明。

> 说明：如果数据库迁移成本过高，也可先在接口层写死 `resource_type` 并部分计算 `process_message`，但优先建议补字段，因为成本较低且表达清晰。

## 5.2 新增对象详情聚合接口

建议在 `backend/app/main.py` 中新增：

- `GET /assets/{asset_id}`

该接口不应只是回传原始表字段，而应返回面向详情页的聚合结构，至少包含：

### 对象摘要
- `id`
- `identifier`（如 `asset-{id}`）
- `title`
- `resource_type`
- `resource_type_label`
- `status`
- `process_message`
- `created_at`

### 文件信息
- `filename`
- `file_path`
- `file_size`
- `mime_type`

### 技术元数据
- 图像宽高
- `fixity_sha256`
- `ingest_method`
- `conversion_method`
- `original_file_path`
- 其他已提取字段

### 访问结果
- `manifest_url`
- 是否可预览

### 输出入口
- 原文件下载 URL
- BagIt 下载 URL

## 5.3 新增原文件下载接口

建议新增：

- `GET /assets/{asset_id}/download`

目的：

- 让第一阶段输出动作更直观；
- 避免把 BagIt 作为唯一输出方式；
- 更符合“对象交付”而不是“仅长期保存打包”的第一阶段叙事。

## 5.4 统一状态表达

当前阶段建议统一围绕以下状态：

- `processing`
- `ready`
- `error`

并通过 `process_message` 显示更具体说明，例如：

- “等待处理”
- “正在将 PSB 转换为 BigTIFF”
- “处理完成，可预览”
- “转换失败：xxx”

该方案比引入复杂工作流状态更稳妥，足够支撑当前演示需求。

---

## 6. 前端改动方案

## 6.1 新增对象详情组件

建议新增：

- `frontend/src/components/AssetDetail.tsx`

这是本轮最关键的前端补充。

建议页面分为 5 个区块：

### 区块 A：对象摘要
显示：
- 标题
- 标识
- 类型（二维文物图片资源对象）
- 当前状态
- 创建时间

### 区块 B：文件信息
显示：
- 文件名
- MIME Type
- 文件大小
- 当前文件路径（开发阶段可显示，后续可隐藏或摘要化）

### 区块 C：技术元数据
显示：
- 宽度 / 高度
- SHA256
- ingest method
- conversion method
- original file path
- 其他可读字段

### 区块 D：访问结果
提供：
- 打开 IIIF Manifest
- 打开 Mirador 预览

### 区块 E：输出动作
提供：
- 下载当前文件
- 下载 BagIt 包

## 6.2 App 中增加“详情态”

当前前端未使用完整 React Router，而是通过菜单切换模块。为了最小改动，本轮建议：

- 在 `App.tsx` 中增加 `selectedAssetId` 状态；
- 当 `selectedAssetId !== null` 时，渲染详情页；
- 列表页和上传成功路径都可进入详情页。

这样可避免本轮引入完整路由系统，提高实现稳定性。

## 6.3 调整资产列表操作入口

建议将当前“查看”主按钮改造为：

- `详情`
- `预览`
- `下载`
- `删除`

即：

- 详情页成为主入口；
- Mirador 预览作为详情页内/外均可触达的辅助入口。

## 6.4 调整 IngestDemo 成功态

建议在 `IngestDemo` 中增加：

- `onOpenAssetDetail(assetId)` 回调；
- 上传成功后新增“查看对象详情”按钮；
- 甚至可将“查看对象详情”作为主后续动作。

这样可把演示路径自然地从“上传成功提示”延伸到“对象详情页”。

---

## 7. 推荐实施顺序

### 第 1 步：后端接口补齐
- 轻量增强 `Asset` 字段；
- 新增 `GET /assets/{id}`；
- 新增 `GET /assets/{id}/download`；
- 统一设置/补充 `process_message`。

### 第 2 步：前端详情页落地
- 新增 `AssetDetail.tsx`；
- App 中加入 `selectedAssetId` 逻辑；
- 列表页支持进入详情页。

### 第 3 步：打通上传后进入详情页
- 为 `IngestDemo` 增加对象详情跳转；
- 上传成功后直接进入对象详情。

### 第 4 步：静态验证与演示路径复核
- 编译前端；
- 校验后端导入与接口结构；
- 从“上传 → 详情 → 预览 → 下载”走查一次代码路径。

---

## 8. 完成后的阶段成果

本轮完成后，MDAMS 将从当前的：

> 上传 / 列表 / Mirador / 下载的功能点集合

收束为：

> **单件二维文物图片资源对象闭环原型（第一阶段）**

其价值在于：

- 更符合当前项目“资源对象而非单文件”的方向；
- 演示时更容易解释系统价值；
- 后续可以继续在此基础上扩展状态表达、对象关系和类型差异；
- 不需要立即冒险进行大规模架构重构。

---

## 9. 与下一阶段的关系

本轮实现完成后，下一步最自然的延伸方向将是：

1. 将当前 `Asset` 详情页进一步演化为更一般的对象详情页；
2. 增强状态表达与失败反馈；
3. 逐步梳理对象—文件—衍生表示—元数据之间的关系；
4. 在不破坏当前可演示闭环的前提下，为未来多类型对象扩展做准备。

因此，本轮不是终点，而是：

> 从技术 PoC 向业务可理解原型过渡的关键一步。
