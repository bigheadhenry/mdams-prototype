# MDAMS 资源对象详情第二轮实现计划：衍生文件关系与对象结构整理

## 1. 本轮目标

在第一轮“对象详情页与闭环收口”已经完成的基础上，本轮目标不是继续横向扩功能，而是把当前详情页从“字段展示页”进一步整理为更接近**资源对象表达层**的结构。

本轮聚焦两件事：

1. 把当前对象内部的文件关系表达出来；
2. 把访问/输出入口按语义重新组织，而不是继续作为若干并列按钮。

本轮仍坚持最小改动原则，不做大规模数据模型重构，不引入多表对象体系。

---

## 2. 当前问题

虽然现有 `GET /assets/{id}` 与 `AssetDetail` 已能展示对象摘要、技术元数据和下载入口，但从对象表达角度看，仍有几个明显不足：

- 当前文件信息区只是在展示“一个当前文件”，还没有表达该对象可能同时包含原始文件、当前有效文件、衍生文件、导出包等不同角色文件；
- 访问（IIIF / Mirador）与输出（原文件下载 / BagIt 下载）还只是按钮集合，语义区分不够清楚；
- 当前页面更像调试信息整合页，而不是对象结构页；
- 后续如果扩展到 PSB、古籍扫描、3D 等类型，现有详情结构不够稳定。

因此，本轮重点不是新增更多功能，而是先把对象详情整理成一个可扩展骨架。

---

## 3. 本轮实现原则

### 3.1 不改动核心链路语义

以下内容保持不变：

- 现有 `Asset` 仍作为当前阶段资源对象承载记录；
- 现有 `GET /assets/{id}` 仍为详情主接口；
- 现有 IIIF / Manifest / Mirador / BagIt 链路不改语义；
- 现有数据库不引入复杂迁移。

### 3.2 优先通过聚合结构改进对象表达

本轮优先在接口返回结构与前端展示结构上增强：

- 通过详情接口新增 `file_roles` / `representations` / `access_paths` / `output_actions` 等聚合结构；
- 前端据此按对象视角重组页面，而不是继续直接平铺字段。

### 3.3 先稳定二维图像对象表达

本轮仍只面向：

> 单件二维文物图片资源对象

但输出结构应为将来更多资源类型预留扩展位。

---

## 4. 后端改动方案

## 4.1 增强详情接口结构

在 `GET /assets/{asset_id}` 中保留现有核心字段，同时新增以下聚合区块：

### A. `structure`
用于描述对象内部文件关系，至少包含：

- `primary_file`：当前系统视为主要访问/管理对象的文件
- `original_file`：若存在原始源文件，则单独表达
- `derivatives`：衍生文件列表（当前阶段至少为可空列表）
- `packaging`：与导出/长期保存打包相关的结构说明

### B. `access_paths`
用于表达“访问”而非“输出”：

- `manifest`
- `mirador_preview`
- `preview_enabled`

### C. `output_actions`
用于表达“交付/导出”：

- `download_current_file`
- `download_bag`

### D. `status_info`
对当前状态做更稳定的前端友好聚合：

- `code`
- `label`
- `message`
- `preview_ready`
- `has_error`

## 4.2 文件关系判定规则

当前阶段采用轻量规则：

- `asset.file_path` 指向当前主要文件；
- 若 `metadata_info.original_file_path` 存在且与 `asset.file_path` 不同，则视为原始文件；
- 若二者相同或缺失，则说明当前对象暂无独立“原始文件/衍生文件”区分；
- `BagIt` 不作为对象内部文件，而作为输出/打包动作表达。

## 4.3 衍生文件最小表达

本轮不新建衍生文件表，只先在接口中给出可扩展列表：

- 若存在已知转换关系（如 PSB -> TIFF），则把当前主文件可同时标注为 `access_master` 或 `converted_master`；
- `derivatives` 先允许为空；
- 后续再决定是否落真实衍生文件记录。

---

## 5. 前端改动方案

## 5.1 重组详情页结构

将 `AssetDetail.tsx` 从现有的平铺区块，调整为更稳定的对象视图，建议区块如下：

### 区块 A：对象摘要
- 标题
- 对象标识
- 类型
- 状态
- 创建时间

### 区块 B：对象结构 / 文件关系
- 当前主文件
- 原始文件（如存在）
- 衍生文件（如存在）
- 当前说明文字（例如“当前对象由原始文件与当前访问文件构成”）

### 区块 C：技术元数据
- 宽度 / 高度
- SHA256
- 入库方式
- 转换方式
- 错误信息（如有）

### 区块 D：访问路径
- Mirador 预览
- IIIF Manifest
- 是否可预览

### 区块 E：输出动作
- 下载当前文件
- 下载 BagIt 包

## 5.2 增强状态可读性

前端显示不再只依赖原始 `ready/processing/error`，而是优先使用 `status_info`：

- 中文标签
- 说明信息
- 是否可预览
- 是否错误

## 5.3 不引入完整路由重构

本轮仍不改造为完整路由系统，继续沿用当前最小改动方案。

---

## 6. 验证目标

本轮完成后，应至少验证：

1. 详情接口返回新的对象结构聚合字段；
2. 前端详情页能展示“当前主文件 / 原始文件 / 衍生文件占位”；
3. 访问路径与输出动作在界面上分区明确；
4. 对于普通上传图像与 SIP 入库图像，页面结构都能稳定显示；
5. 原有下载、Manifest、BagIt 功能不被破坏。

---

## 7. 本轮成果预期

本轮完成后，MDAMS 的对象详情将从：

> 一个可用的对象详情页

进一步提升为：

> 一个开始具备“对象内部结构表达能力”的资源对象详情原型

这将为后续继续接入：

- 衍生文件真实建模
- 处理事件留痕
- 多资源类型对象差异化表达
- 更清晰的访问/输出边界

打下更稳的基础。
