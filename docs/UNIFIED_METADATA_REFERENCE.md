# MDAMS 统一元数据参考

## 1. 文档目的

本文档用于规范 MDAMS 项目中的统一元数据思路，作为后续研究、开发、接入外部系统、设计数据库结构和接口返回结构的参考基线。

本文档的目标不是要求所有模态、所有系统使用完全相同的字段集合，而是明确：

- 哪些字段属于跨系统公共元数据；
- 哪些字段属于模态专有元数据；
- 哪些字段适合作为展示层字段；
- 哪些字段适合作为关系语义字段；
- 统一元数据体系在项目中如何演进与维护。

---

## 2. 总体原则

MDAMS 的统一元数据体系遵循以下原则：

### 2.1 统一最小公共元数据，而不是统一全部字段

项目不追求把所有资源、所有模态、所有业务字段都塞进同一套结构中。更合理的方式是：

- 顶层维护一份最小但稳定的公共元数据；
- 子系统维护各自的模态专有字段；
- 通过统一资源 ID 和映射关系把两者关联起来。

### 2.2 顶层负责公共语义，底层负责专业语义

- 顶层聚合层负责统一发现、统一检索、统一展示、统一权限视图；
- 底层模态系统负责文件处理、专业工作流和模态专有数据。

### 2.3 统一元数据必须可扩展

统一元数据不是固定不变的。随着项目发展，它应当允许：

- 新增字段；
- 新增模态；
- 新增关系；
- 新增同步策略；
- 新增外部系统接入。

---

## 3. 权威标准参考

MDAMS 统一元数据体系建议优先参考以下三类标准：

### 3.1 Dublin Core

Dublin Core 适合作为公共元数据最小集的参考。

它强调：

- 使用少量稳定字段描述资源；
- 可与其他词表组合使用；
- 适合跨系统、跨领域的基础描述。

适合在 MDAMS 中承担：

- 标题
- 描述
- 创建者
- 主题
- 权利
- 时间
- 来源
- 标识符

等公共字段的参考基础。

### 3.2 CIDOC CRM

CIDOC CRM 适合作为文化遗产语义关系建模的参考。

它强调：

- 对象、事件、人物、地点、来源之间的关系；
- 文化遗产信息的语义整合；
- 跨来源、跨系统的数据融合。

适合在 MDAMS 中承担：

- 资源之间的关系建模；
- 资源与来源系统的映射；
- 资源与人物、事件、地点的关联。

### 3.3 IIIF Presentation API

IIIF Presentation API 适合作为展示层和浏览层参考。

它强调：

- 面向用户的结构化展示；
- 资源视图、页面顺序、导航关系；
- 与 Image API / Viewer 组合形成可浏览体验。

适合在 MDAMS 中承担：

- 对象结构展示；
- 资源浏览组织；
- 预览和可视化访问层表达。

## 4. MDAMS 推荐的统一元数据分层

MDAMS 建议将元数据分成四层。

### 4.1 公共描述层

这一层保存跨系统通用、可被统一发现和检索的字段。

建议字段包括：

- `global_resource_id`
- `source_system`
- `source_object_id`
- `resource_type`
- `title`
- `summary`
- `keywords`
- `creators`
- `rights`
- `access_level`
- `status`
- `preview_url`
- `detail_url`
- `updated_at`

这一层的职责是让资源能够被统一识别、统一检索和统一展示。

### 4.2 模态专有层

这一层保存模态独有的深字段。

例如：

- 图像：分辨率、色彩空间、IIIF 参数、切片信息
- 视频：时长、编码、帧率、关键帧
- 音频：采样率、声道、波形
- 文档：页数、OCR、版本
- 3D：面数、材质、LOD

这一层不要求统一到所有系统共用一套字段。

### 4.3 关系语义层

这一层用于描述资源之间、资源与人物之间、资源与事件之间、资源与来源之间的关系。

建议字段或概念包括：

- 关联对象
- 关联事件
- 关联地点
- 来源系统映射
- 原始资源 / 派生资源
- 版本关系
- 复制关系
- 引用关系

这一层适合参考 CIDOC CRM 的思路。

### 4.4 展示组织层

这一层用于面向用户的浏览和展示。

建议字段或概念包括：

- 展示标题
- 展示摘要
- 预览入口
- 详情入口
- IIIF Manifest
- 浏览顺序
- 页面/视图结构

这一层适合参考 IIIF Presentation API 的思路。

## 5. 统一资源 ID 规则

MDAMS 建议使用统一资源 ID 作为跨系统组织的主线。

### 5.1 统一资源 ID 的作用

统一资源 ID 用于：

- 跨系统映射；
- 聚合检索；
- 资源关联；
- 统一展示；
- 权限协同；
- 同步更新。

### 5.2 推荐同时保存的标识字段

- `global_resource_id`：平台全局资源 ID
- `source_system`：来源系统标识
- `source_object_id`：来源系统对象 ID
- `local_object_id`：子系统本地对象 ID（如有）

### 5.3 设计原则

- `global_resource_id` 是跨系统唯一主键；
- `source_system + source_object_id` 用于回源定位；
- 子系统可保留本地 ID，但不能替代全局 ID；
- 如存在共享字段，必须明确谁是主数据源。

---

## 6. 顶层系统与子系统的职责

### 6.1 顶层系统保存什么

顶层系统建议保存公共描述层与聚合视图所需字段，包括：

- 全局资源标识；
- 公共元数据；
- 统一权限视图；
- 统一入口信息；
- 关联关系摘要；
- 统一状态。

### 6.2 子系统保存什么

子系统建议保存：

- 全局资源标识；
- 本地对象标识；
- 模态专有字段；
- 文件路径与处理信息；
- 本地工作流状态；
- 本地展示所需最小字段。

### 6.3 共享字段怎么处理

如果某些字段顶层和子系统都保存，应明确：

- 谁是主数据源；
- 谁是同步副本；
- 谁负责更新；
- 同步失败如何处理；
- 不一致时以谁为准。

## 7. MDAMS 当前已形成的统一元数据雏形

结合当前项目代码与页面结构，MDAMS 已经具备部分统一元数据雏形，包括：

- `resource_type`
- `status`
- `process_message`
- `created_at`
- `technical_metadata`
- `process_timeline`
- `structure`
- `access_paths`
- `output_actions`

这些字段目前主要服务于图像资源对象，但它们已经体现出：

- 对象化表达；
- 处理轨迹表达；
- 访问与输出分离；
- 资源结构与技术元数据分层。

后续这些字段可进一步抽象为跨模态对象表达的基础。

---

## 8. 未来演进方向

随着项目发展，统一元数据体系应继续向以下方向演进：

### 8.1 统一字段收敛

持续明确哪些字段应成为公共字段，哪些字段应留在模态专有层。

### 8.2 关系建模增强

逐步丰富资源与资源、资源与人物、资源与事件、资源与地点之间的关系表达。

### 8.3 接入映射标准化

为外部系统定义统一接入映射格式，减少系统接入成本。

### 8.4 展示层和语义层分离

继续坚持：

- 展示层服务于浏览与人类理解；
- 语义层服务于检索、关联和系统集成。

### 8.5 AI 可读性增强

新的元数据设计应尽量保持：

- 字段名明确；
- 语义稳定；
- 关系清晰；
- 层次分明；
- 易于模型和开发者共同理解。

---

## 9. 当前阶段建议

当前阶段建议优先做到：

1. 明确公共字段最小集；
2. 明确顶层/子系统职责边界；
3. 为统一资源 ID 建立稳定映射；
4. 逐步抽象图像对象详情页；
5. 在新增模态或新增系统时，持续沿用本文件中的原则。
