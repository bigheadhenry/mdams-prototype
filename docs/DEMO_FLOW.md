# MDAMS Prototype 标准演示流程

## 1. 文档目标

本文档用于定义 **MDAMS Prototype** 的标准演示流程，目标是让开发、联调、汇报和对外展示时都能按照同一套路径完成演示。

本流程重点覆盖以下主链路：

- 服务启动
- 资产上传
- 入库验证
- 元数据提取验证
- 转换任务验证
- IIIF Manifest 验证
- Mirador 预览验证
- 下载/导出验证

本文档不追求覆盖所有边界情况，而是强调：

> **用一条稳定、可复核的路径验证系统当前最核心的价值。**

---

## 2. 演示前提

开始演示前，应确认以下前提成立：

### 2.1 基础环境已启动

建议至少确认以下服务已启动且状态正常：

- frontend
- backend
- db
- redis
- worker
- cantaloupe
- nginx（如当前部署链路依赖）

可使用：

```bash
docker compose ps
```

如需查看配置展开结果，可使用：

```bash
docker compose config
```

### 2.2 关键访问入口可用

至少确认以下入口可访问：

- 前端首页
- 后端 API 基础入口
- IIIF 相关访问路径

如当前部署约定仍为：

- 前端入口：`http://<HOST>:3000`
- API 前缀：`http://<HOST>:3000/api`
- IIIF 前缀：`http://<HOST>:3000/iiif/2`

则演示前应先完成一次基本访问检查。

### 2.3 准备演示样本

建议至少准备以下样本：

1. **普通图像样本**：用于快速验证基础流程
2. **较大图像样本**：用于验证处理与预览链路
3. **超大图像样本（PSB / TIFF）**：用于验证核心差异化能力
4. **带基础元数据样本**：用于验证元数据提取结果

---

## 3. 标准演示主流程

## 步骤 1：确认系统已成功启动

### 操作

执行：

```bash
docker compose ps
```

### 预期结果

- 关键服务处于运行状态
- 没有明显反复重启的容器
- 前端、后端、worker、cantaloupe 至少处于可用状态

### 演示说明建议

可向观众说明：

- 系统采用容器化部署
- 上传、处理、预览涉及多个服务协作
- 当前演示重点是完整链路而非单一页面

---

## 步骤 2：打开前端并进入资产管理入口

### 操作

在浏览器打开前端地址，例如：

```text
http://<HOST>:3000
```

进入资产列表或上传入口页面。

### 预期结果

- 页面可正常加载
- 前端没有明显白屏或接口报错
- 可以进入上传相关界面

### 重点说明

此步骤验证的是：

- 前端服务可访问
- 前端与后端 API 基本联通
- 当前部署入口路径正确

---

## 步骤 3：上传一个标准样本文件

### 操作

在前端上传一个准备好的演示样本。

建议优先顺序：

1. 先用普通图像快速验证流程
2. 再用超大图像样本展示差异化能力

### 预期结果

- 上传请求成功
- 前端显示成功反馈或进入后续处理状态
- 后端可看到新资产记录或上传记录

### 重点说明

可向观众说明：

- 系统不是单纯文件存储，而是面向数字资产处理链路
- 上传之后不仅保存文件，还会进入后续入库/处理流程

---

## 步骤 4：验证资产已入库

### 操作

在前端资产列表中查看新记录，或直接通过 API 检查。

例如可检查资产列表接口：

```text
GET /api/assets
```

### 预期结果

- 新上传资产出现在列表中
- 至少可看到资产基础信息
- 资产 ID 或唯一标识可用于后续查询

### 重点说明

此步骤主要验证：

- 上传结果已进入资产管理视图
- 后端数据库写入与列表查询链路正常

---

## 步骤 5：验证元数据提取结果

### 操作

查看新资产的元数据字段或详情页返回信息。

重点检查：

- 文件类型
- 文件大小
- 图像尺寸
- 基础技术元数据
- 已提取的 Exif / IPTC 等信息（若样本包含）

### 预期结果

- 至少可以看到部分基础技术元数据
- 元数据字段与样本内容基本一致
- 无明显空值异常或结构错误

### 重点说明

此步骤用于展示：

- 系统具备对数字对象做“处理与识别”的能力
- 不是只做文件上传展示

---

## 步骤 6：验证异步处理/转换任务

### 操作

对于需要转换的样本，检查处理任务状态。

可从以下角度观察：

- 前端状态字段
- 后端任务状态接口（如已有）
- worker 日志
- 转换结果文件是否出现

### 预期结果

- 转换任务被成功触发
- worker 正常执行
- 转换完成后状态可见
- 如涉及 PSB → BigTIFF，能看到预期衍生物结果

### 重点说明

这是项目最关键的差异化能力之一，建议在演示中明确强调：

- 面向超大图像资源的处理能力
- 异步任务与预览资源生成能力

---

## 步骤 7：验证 IIIF Manifest

### 操作

获取该资产对应的 manifest 地址，并在浏览器中直接访问。

### 预期结果

- manifest 可访问
- 返回结构符合 IIIF Manifest 的基本预期
- 其中包含正确的资源引用关系

### 建议检查点

- manifest URL 是否正确
- 资源标识是否正确
- 图像服务地址是否指向当前可访问的 IIIF 路径

### 重点说明

此步骤展示的是：

- 系统与 IIIF 标准的结合能力
- 后续对外共享、标准化展示的基础能力

---

## 步骤 8：在 Mirador 中完成预览

### 操作

通过前端详情页或专门预览入口，在 Mirador 中打开该资产。

### 预期结果

- Mirador 成功加载 manifest
- 图像能够缩放、平移
- IIIF 服务返回图像切片正常

### 重点说明

这是最直观的展示环节，建议明确强调：

- 预览不是直接加载完整大图，而是基于 IIIF 服务进行访问
- 该能力对超大图像场景非常关键

---

## 步骤 9：验证下载/导出链路

### 操作

执行系统提供的下载或导出功能，例如 BagIt ZIP 下载。

### 预期结果

- 下载链接可访问
- 导出文件生成成功
- 导出内容与资产记录一致

### 重点说明

此步骤展示的是：

- 系统具备从“入库/预览”延伸到“导出/交付”的能力

---

## 4. 推荐演示话术结构

可按以下逻辑讲述：

1. **系统定位**：这是一个面向博物馆/展览数字资源场景的技术原型
2. **核心价值**：重点验证超大图像资产的上传、处理、IIIF 化和预览链路
3. **演示顺序**：上传 → 入库 → 元数据 → 转换 → Manifest → Mirador → 下载
4. **关键亮点**：PSB / BigTIFF / IIIF / Docker 化部署
5. **当前边界**：现阶段重点是核心链路与技术可行性，完整业务化能力仍在后续建设中

---

## 5. 演示通过标准

如果以下各项都成立，可认为本次标准演示通过：

- 服务启动正常
- 前端可访问
- 资产上传成功
- 资产列表可见新记录
- 元数据可见
- 异步转换执行成功
- Manifest 可访问
- Mirador 可正常预览
- 下载/导出可执行

---

## 6. 演示失败时优先检查

如果演示中断，建议优先按以下顺序检查：

1. `docker compose ps` 是否显示关键服务正常
2. 前端访问路径是否正确
3. 后端 API 是否可访问
4. worker 是否在执行任务
5. Cantaloupe 是否可访问源图像与衍生图像
6. manifest 中引用的 URL 是否与当前部署入口一致
7. nginx 代理路径是否与 `/api`、`/iiif/2` 约定一致

更详细的排查方法见：

- `docs/TROUBLESHOOTING.md`
