# 数据传输与入库架构

## 1. 核心思路

当前项目的入库策略不是“先压成一个大包再上传”，而是更适合 Web 的分层入库方式。

### 二维影像

二维影像当前采用的是一种分离式 SIP 风格上传：

- 原始文件通过 `multipart/form-data` 上传
- JSON manifest 同步上传
- 后端再做 fixity 校验、元数据提取和入库

### 三维资源

三维资源则采用资源包思路：

- 一个资源对象
- 多个版本
- 每个版本下挂模型、点云、倾斜摄影等文件
- 通过 `web_preview_status` 决定是否进入前端展示

## 2. 二维入库流程

### 2.1 上传

前端提交：

- 原始文件流
- manifest JSON

后端接收后会：

1. 计算服务器端哈希
2. 与客户端 manifest 中的哈希比对
3. 保存物理文件到上传目录
4. 写入数据库元数据
5. 更新资源状态

### 2.2 元数据分层

当前二维资源元数据已经不是单一 JSON，而是分层组织：

- `core`
- `management`
- `technical`
- `profile`
- `raw_metadata`

其中：

- `core` 用于平台统一目录
- `technical` 用于文件和处理信息
- `profile` 用于文物或对象类型语义
- `raw_metadata` 用于保留原始来源字段

### 2.3 入库后的状态

入库完成后，资源会从 `processing` 推进到 `ready` 或可解释的错误状态。

## 3. 二维导出

二维资源支持：

- 原文件下载
- BagIt ZIP 导出
- 申请交付包导出

导出时会保留当前资源文件与必要的元数据说明。

## 4. 三维入库流程

三维资源当前使用版本化资源包方式：

- 原始版
- v1 / v2 / v3
- Web 展示版

每个版本可以挂：

- 模型文件
- 点云文件
- 倾斜摄影图像

并由 `web_preview_status` 决定是否允许 Web 展示。

## 5. 这个架构的意义

这样设计的原因是：

- 既能保留原始数据
- 也能保留平台所需的结构化元数据
- 还能给 IIIF、Mirador、三维 viewer 和申请交付提供统一入口

## 6. 当前结论

当前的入库架构已经不只是“文件上传”，而是：

> **文件接收 + 哈希校验 + 元数据分层 + 资源状态管理 + 导出能力**

后续可以继续补：

- 更完整的校验规则
- 更细的对象映射
- 更严格的导入模板
- 更清楚的失败回溯信息
