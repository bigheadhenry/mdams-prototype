# 对象 Profile 识别规则

本文档定义二维影像子系统在 MDAMS 平台中的对象 profile 识别方式。目标不是把所有二维图片都归成同一类，而是让系统在“没有对象语义时默认保持通用，在出现明确对象字段时再进入具体 profile”。

## 基本原则

1. 平台核心层只放所有资源都成立的字段。
2. 共用管理层放采编和治理信息。
3. 技术影像层放文件、尺寸、校验、转换等信息。
4. 对象 profile 层只在确实存在对象语义时启用。
5. 没有明确对象元数据时，默认落到 `其他`。

## Profile 判定顺序

系统按以下顺序判断 profile：

1. 显式 profile 提示。
2. 识别到对象字段后，按字段命中具体 profile。
3. 没有命中任何对象字段时，归入 `其他`。

显式提示包括：
- `metadata_profile`
- `profile`
- `image_category`
- `影像类别`
- `object_profile`

## 识别规则表

| Profile key | Sheet 名称 | 典型命中字段 | 说明 |
| --- | --- | --- | --- |
| `movable_artifact` | `可移动文物` | `文物号`、`文物名称`、`文物类别`、`文物细类` | 文物图片默认对象类型，只有在检测到对象字段时才启用 |
| `immovable_artifact` | `不可移动文物` | `一级区域`、`二级区域`、`文物建筑名称`、`方位`、`部位一/二/三`、`建筑构件` | 建筑与遗址类资源 |
| `art_photography` | `艺术摄影` | `艺术摄影类型`、`藏品类型`、`所在宫区`、`季节`、`植物`、`动物`、`节气`、`主题`、`文化专题`、`展览专题` | 艺术图像或专题摄影 |
| `business_activity` | `业务活动` | `主要地点`、`主要人物` | 业务过程、活动记录 |
| `panorama` | `全景` | `全景类型`、`位置` | 全景类资源 |
| `ancient_tree` | `古树` | `档案编号`、`植物类型`、`植物名称`、`所在区域`、`具体位置`、`等级` | 古树与树木档案 |
| `archaeology` | `考古` | `考古影像分类` | 考古记录资源 |
| `other` | `其他` | 无 | 默认值，表示未识别到明确对象语义 |

## 元数据分层建议

推荐将二维影像资源组织为以下 5 层：

```json
{
  "core": {},
  "management": {},
  "technical": {},
  "profile": {
    "key": "movable_artifact",
    "label": "可移动文物",
    "sheet": "可移动文物",
    "fields": {}
  },
  "raw_metadata": {}
}
```

### `core`

平台统一层，适合目录、检索、详情入口使用。

### `management`

共用管理元数据，例如：
- `项目类型`
- `项目名称`
- `摄影者/制作者`
- `拍摄/制作时间`
- `影像类别`
- `影像名称`
- `是否为代表影像`
- `影像标签`

### `technical`

技术影像元数据，例如：
- `原始文件名`
- `影像文件名`
- `文件大小`
- `格式名称`
- `格式版本`
- `消息摘要算法`
- `消息摘要`
- `图像宽度`
- `图像高度`
- `色彩空间`
- `原始文件路径`
- `转换方式`

### `profile`

只存当前资源命中的对象 profile 字段。例如可移动文物中的：
- `文物号`
- `文物名称`
- `文物类别`
- `文物细类`
- `管理科组`

### `raw_metadata`

保留来源系统原始字段，用于追溯和后续映射。

## 索引策略

统一检索不只搜 `core`，还应按受控范围搜：

- `core`
- `management`
- `technical`
- `profile.fields`
- 少量受控的 `raw_metadata`

不要把整个原始 JSON 直接暴露成全文检索对象。

## 实际落地约束

- 没有对象号、对象名称等字段时，默认 `other`。
- 检测到对象字段后再进入具体 profile。
- 同一资源可只有技术层和管理层，没有 profile 层。
- 平台目录和统一详情应优先展示 `core.title`。
