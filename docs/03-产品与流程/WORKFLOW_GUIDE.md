# 核心工作流指南

- 最后核对日期：2026-04-06
- 核对范围：`frontend/src/App.tsx`、`backend/app/routers/`、`frontend/src/components/`

## 1. 目标

本文件用于概括当前 MDAMS 原型已经落地的核心业务工作流，帮助读者快速理解系统不是“单页面上传工具”，而是由多条关联链路组成。

## 2. 总体流程图

当前项目可以粗分为五条核心链路：

1. 登录与角色识别
2. 二维资产上传与查看
3. 图像记录拆分工作流
4. 统一平台浏览与跳转
5. 申请、审批与交付

三维资源链路与统一平台链路并行存在。

## 3. 登录与角色识别

当前登录流程：

1. 前端读取 `/api/auth/users` 获取可用测试用户
2. 用户提交用户名和密码到 `/api/auth/login`
3. 后端返回 token 与当前用户上下文
4. 前端保存 token，并继续读取 `/api/auth/context`
5. 前端按权限裁剪菜单与页面入口

当前角色识别不仅影响“能不能看到菜单”，还影响：

- 能否读取资产列表
- 能否进入图像记录工作台
- 能否进入申请管理
- 能否看到责任范围内资源

## 4. 二维资产主链路

当前二维资产主链路如下：

1. 具备 `image.upload` 的用户上传文件
2. 后端创建资产记录并写入挂载目录
3. 后端生成元数据层与基础状态
4. 需要衍生文件时进入 IIIF access 处理
5. 资产在列表页、详情页和 Mirador 中可查看

当前这条链路覆盖：

- 上传
- 列表
- 详情
- 预览图
- Manifest
- Mirador
- 下载

## 5. 图像记录拆分工作流

当前图像记录工作流是项目的一条重要新链路。

### 5.1 元数据录入人员

`image_metadata_entry` 负责：

- 创建图像记录
- 编辑结构化信息
- 提交图像记录
- 退回后继续修订

### 5.2 摄影上传人员

`image_photographer_upload` 负责：

- 查看待上传记录池
- 上传文件
- 与指定记录匹配
- 触发上传后的校验流程

### 5.3 当前意义

这条链路意味着当前系统已经从“先传文件再补说明”演进为“记录和文件可以分离协作”。

## 6. 统一平台链路

统一平台当前是聚合入口，而不是新的独立资源来源。

当前流程：

1. 用户进入统一资源目录
2. 前端请求 `/api/platform/sources` 和 `/api/platform/resources`
3. 平台层从已注册来源适配器聚合资源
4. 用户按条件筛选资源
5. 从目录跳转到统一详情或来源详情

当前统一平台主要用于：

- 跨来源浏览
- 统一筛选
- 统一资源详情跳转

## 7. 三维资源链路

当前三维相关流程如下：

1. 具备 `three_d.view` 或更高权限的用户进入三维管理
2. 三维操作人员上传单文件或多文件资源包
3. 后端保存对象、版本和文件角色信息
4. 前端展示三维详情和查看摘要
5. 统一平台可继续聚合三维来源

当前三维链路已经具备：

- 对象与版本概念
- 多文件角色区分
- Web 查看入口
- 平台聚合接入

## 8. 申请、审批与交付链路

当前申请相关流程如下：

1. `resource_user` 浏览资源
2. 用户把目标资源加入申请车
3. 前端提交申请单到 `/api/applications`
4. `application_reviewer` 进入申请管理页面
5. 审批通过或拒绝
6. 审批通过后可导出交付包

这条链路当前已经形成了从浏览到导出的最小闭环。

## 9. 当前工作流与菜单的关系

前端菜单与工作流的对应关系大致如下：

| 菜单 | 对应工作流 |
| :--- | :--- |
| 仪表盘 | 登录后总览 |
| 二维资源 | 二维资产浏览与详情 |
| 申请车 | 申请提交 |
| 入库处理 | 二维上传与入库操作 |
| 统一资源目录 | 平台聚合浏览 |
| 统一资源详情 | 平台详情 |
| 三维管理 | 三维对象与版本 |
| 申请管理 | 审批与导出 |
| 图像记录工作台 | 元数据录入与摄影上传协作 |

## 10. 当前边界

当前工作流虽然已成型，但仍有这些边界：

- 平台统一检索还不完整
- 图像记录与正式入库的状态治理还可继续增强
- IIIF 访问控制还需要继续收口
- 三维治理与兼容性仍需完善
- 申请流的更细粒度规则仍可继续细化
