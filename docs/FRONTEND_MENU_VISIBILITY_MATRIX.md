## 设计说明

### 1. 总览默认对所有馆内角色开放

总览页不是业务敏感入口，它更适合作为当前登录用户的角色摘要页和系统导航首页。

### 2. 统一资源目录和统一资源详情属于平台基础入口

只要一个角色具备资源浏览能力，就应允许其进入统一目录与统一详情。区别不在于菜单是否可见，而在于后端返回什么资源、前端是否开放进一步动作。

### 3. 申请车只对资源浏览/申请人员开放

当前版本里，申请功能仍定位为二维影像子系统的业务子模块，因此申请车入口只对 `resource_user` 和 `system_admin` 可见。

### 4. 申请管理只对审批角色开放

申请管理属于审批与交付模块，不应暴露给普通浏览/申请用户。

### 5. 三维管理与三维浏览后续可继续拆分

当前原型阶段，`three_d_operator`、`collection_owner` 和 `resource_user` 都可进入三维模块，但后续建议继续细化为：

- 三维管理页
- 三维浏览页
- 三维对象详情页

这样可以把“上传维护”和“查看使用”进一步拆开。

## 第一阶段实现原则

1. 菜单先裁剪
先让不同角色看到不同菜单，这是用户最直接能感知到的差异。

2. 页面再裁剪
即使菜单可见，也要根据动作权限控制按钮是否出现，例如删除、审批、导出。

3. 接口必须兜底
即使前端已经隐藏菜单和按钮，后端仍然必须进行权限校验。

## 当前代码映射

当前项目中，第一版菜单矩阵已对应到前端权限骨架：

后续如果新增菜单，应先同步更新本文档，再更新前端权限映射。
