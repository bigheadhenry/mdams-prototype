# Project Brief

## Problem
- 博物馆/展陈场景中的数字资源管理往往落在多个分散环节之间：采集、上传、校验、处理、预览、申请、交付、长期保存意识彼此脱节。
- 现有项目实践常在两端摇摆：要么只是文件上传下载工具，要么一开始就试图覆盖完整生产级 DAMS/藏品管理体系，导致原型阶段难以解释、验证和持续推进。
- MDAMS 的核心问题不是“做尽可能多功能”，而是用有限但扎实的实现，证明一个以数字资产为中心、可演示、可评估、可研究表达的原型路径。

## Users
- 馆内数字资源管理与采集人员
- 图像记录录入、摄影上传、审核、资源管理等角色
- 三维资源管理与生产链路相关角色
- 需要利用申请、审批和交付导出的业务人员
- 研究写作主体：需要把工程实现转化为论文/报告材料的项目作者

## Prototype Goal
- 证明系统可以围绕数字资产形成稳定主链路：采集/上传、校验、处理、访问、导出。
- 证明二维影像、图像记录工作流、统一平台、三维资源、申请交付、权限范围控制可以在一个一致的原型框架中共存。
- 证明 IIIF、BagIt、OAIS、PREMIS、NISO Z39.87 等标准/框架可以通过分层、选择性的方式进入实现，而不是做不实的“全面合规”宣称。
- 为后续论文提供可直接复用的事实、决策、评估与局限材料。

## Paper Direction
- 设计科学 / 研究型原型（design science / prototype-oriented research）
- 面向博物馆/展陈数字资源场景的实践驱动系统设计反思
- 一个“具有保存意识的数字资产管理原型”案例，而不是完整生产系统说明
- 重点论证：为什么围绕数字资产组织系统、为什么选择性对齐标准、如何以有限实现形成研究价值

## Boundary
- 当前目标不是完整生产级 DAMS，也不是完整馆藏管理系统。
- 当前权限模型可演示、可测试，但不是完整身份治理体系。
- 当前统一平台可聚合二维与三维来源，但统一检索和更多来源接入仍在后续阶段。
- 当前长期保存、审计、事件 formalization、迁移与监控能力仍不完整。
- AI 能力当前定位为辅助交互与验证入口，不作为系统主贡献点。

## Current Repository Facts
- 仓库已有真实前后端实现、Docker Compose 部署结构、后端测试和前端 Playwright 回归。
- 已有稳定模块包括：二维影像、ImageRecord 工作流、统一平台、三维资源、申请交付、认证与角色范围控制、Mirador AI 面板。
- 当前最强研究表达仍应围绕“主链路”和“对象模型”，而不是继续横向扩展功能面。
