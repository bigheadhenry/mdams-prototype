# Paper Argument

## Main Research Questions
- 在博物馆/展陈数字资源场景下，一个数字资产管理原型最小需要支持哪些核心工作流？
- MDAMS 应把什么视为系统中的核心管理对象，才能同时支撑工程实现与研究表达？
- IIIF、BagIt、OAIS、PREMIS、NISO Z39.87 等标准/框架应如何以分层、选择性的方式进入原型？
- 一个并非完整 OAIS 仓储的系统，如何仍体现“保存导向”而不是普通上传下载平台？

## Core Claim
- MDAMS 的研究价值不在于功能堆叠，而在于它用一个真实、可运行、可演示的原型，展示了如何围绕数字资产组织采集、校验、处理、访问与导出主链路，并以选择性标准对齐形成可解释的系统设计。

## Contribution Candidates
1. 提出一个以数字资产为核心对象的研究型原型框架，用于连接二维影像、三维资源、工作流、访问与导出。
2. 给出一种“选择性标准对齐”的实践路径：IIIF 和 BagIt 作为直接实现层，OAIS 作为概念框架层，PREMIS 与 NISO Z39.87 作为部分对齐层。
3. 通过真实代码、部署路径、回归测试和可演示工作流，提供一个兼具工程可信度与研究可解释性的案例。
4. 展示如何把研究文档、设计决策、评估框架与实现同步，避免论文写作脱离原型实际状态。
5. 以最小事件边界和 proto-event 词表，说明系统如何逐步形成可追踪、可解释的保存与工作流证据层。

## Writing Orientation
- 实践驱动的系统设计反思
- 研究型原型 / 设计科学取向
- 面向文化遗产数字资源管理与保存意识的案例研究表达

## Argument Structure
1. 现实问题：馆内数字资源管理需要的不只是上传工具，也不是一开始就做完整生产级系统
2. 设计选择：围绕数字资产组织主链路，而不是围绕孤立文件或无限扩张功能面
3. 标准策略：采用分层、选择性标准对齐，而不是笼统宣称“遵循多个标准”
4. 实现证据：以二维、图像记录、统一平台、三维、申请交付、权限与测试为支撑
5. 评估方式：看是否有稳定主链路、清晰对象模型、合理标准映射和真实工程可信度
6. 局限与边界：当前仍是原型，不是完整生产系统，也未完成 formal preservation stack

## Available Evidence
- 代码实现：前后端模块、平台适配器、三维子系统、AI 面板
- 测试证据：后端 `pytest` 与前端 `Playwright`
- 部署证据：Docker Compose、Cantaloupe、Redis、PostgreSQL、Celery
- 文档证据：项目状态、工作日志、研究问题、评估框架、设计决策、标准映射

## Argument Boundary
- 不宣称 MDAMS 已实现完整 OAIS、完整 PREMIS、完整 NISO Z39.87 或完整生产级权限治理
- 不把 AI 面板写成系统主贡献，它更适合作为交互增强与验证案例
- 不把三维能力夸大为完整 3D 保存体系，但应承认其已经形成真实实现证据
- 不以功能数量作为主要评价标准，而以问题回应、工作流稳定性和研究可解释性为主

## Current Writing Risk
- 研究线个别文档仍停留在较早阶段的系统事实，如果不及时对齐，会削弱论文论证的可信度
