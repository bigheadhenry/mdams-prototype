# MDAMS Prototype 项目说明书（给未来的自己，也给后来接手的人）

这个项目像一个正在搭建中的“数字文物后台”。它不是那种只会上传文件的简陋样机，也不是一上来就追求面面俱到的大而全平台，而是在真实业务链路上，一块一块把“藏品资源怎么进来、怎么描述、怎么预览、怎么申请使用、怎么按权限看见”这些关键环节搭起来。

如果把它想象成一座博物馆后台，`frontend/` 是前台接待大厅，负责让人看得见、点得动；`backend/app/routers/` 像一排窗口，不同窗口分别办理图片资产、二维著录、三维资源、利用申请、统一目录、AI 辅助这些事项；`backend/app/services/` 则更像后台业务科室，真正处理规则、拼装元数据、生成衍生文件、组织 IIIF 访问、做权限判断。数据库里的 `Asset`、`ImageRecord`、`Application`、`ThreeDAsset` 这些模型，就是馆里那一份份可追踪、可回溯的业务档案。

这个项目技术上选了一个很实用的组合：前端用 React + Vite + TypeScript，后端用 FastAPI + SQLAlchemy，异步任务用 Celery + Redis，高分辨率图像访问交给 Cantaloupe IIIF。这个搭配的好处，不是“潮”，而是职责边界很清楚。FastAPI 负责接口与编排，SQLAlchemy 管结构化数据，Celery 把那些慢活累活挪到后台，IIIF 服务器负责把超大图切片、缩放、按需供图。好工程很多时候不是炫技，而是让每个组件做它最擅长的事，像接力赛一样把球传稳。

代码结构里最值得学习的一点，是它在努力把“资产文件”和“业务记录”分开。图片文件本身属于 `Asset`，而拍摄登记、编目语境、字段校验、流程状态这些属于 `ImageRecord`。这有点像“照片”与“照片背后的档案卡”不是同一件东西。这样设计以后，系统可以分别处理文件技术属性和业务语义属性，后期不管是换存储、换展示方式，还是补更多元数据层，都不会把整条链路绞成一团。

这次我做的改动，核心是把原本独立的 `Face-V2.0` 项目进一步“瘦身”，塞进 `mdams-prototype` 里，变成平台内部可选的一块能力，而不是一整个外部系统。原来 `mdams` 其实已经长出了半条链路：图片著录绑定后，会给“业务活动”类记录挂一个待识别状态，再通过 Celery 异步任务去调用人脸识别服务，最后把结果回写到 `raw_metadata.face_recognition` 和 `profile.fields.main_person`。这说明平台流程已经懂“什么时候该识别、识别后写到哪”，但它还把识别执行器放在平台外面。

真正的精简不是把旧项目整个复制过来，而是像收拾搬家行李一样，先问一句：哪些东西是每天都要用的，哪些只是“看起来以后可能会用”？于是我只保留了最小闭环：

- 模型加载
- 人脸检测与特征提取
- 读取旧项目沉淀下来的 `meta.json + embeddings.pkl` 索引
- 用聚类中心做匹配，返回“识别到了谁”

被明确丢掉的部分也很重要：独立静态前端、整库扫描页面、聚类重命名/合并管理接口、单独的数据目录 UI。这些东西在独立项目里很合理，但在 `mdams` 里会造成“双后台并存”的困惑。平台真正需要的是一个安静、可嵌入、最好不引人注意的“识别引擎”，而不是第二个小系统。

现在的人脸识别策略也更像工程化过渡，而不是赌一条路走到底。配置里新增了 `FACE_RECOGNITION_PROVIDER`，支持：

- `local`：优先用内置本地运行时
- `remote`：继续调用外部服务
- `auto`：先试本地，失败再回退远程

这背后的经验很值得记住。迁移老系统时，最危险的不是“代码写不出来”，而是“一刀切”。给自己留一条平滑回退路径，系统就不会因为一组模型没放对、一个容器没起稳，就把整条业务流程拖死。

这次集成里还有几个典型坑，值得以后反复提醒自己：

1. Docker 里的环境变量不是你在 `.env` 里写了就自动进容器。`docker-compose.yml` 如果没显式把变量传给 `backend` 和 `celery_worker`，应用代码里就等于没配。
2. 把模型放在仓库根目录，不等于容器里也看得到。因为这里的后端镜像构建上下文是 `./backend`，所以真正安全的做法，是把运行时目录放到 `backend/runtime/face_recognition/` 下面，或者另行挂载。
3. 本地推理依赖一定要懒加载。要是 `insightface`、`onnxruntime`、`opencv` 在模块 import 时就硬加载，哪怕功能默认是关闭的，整个后端也可能直接起不来。
4. 旧系统的数据格式要尽量“兼容读”，不要上来就强迫迁移。能直接吃掉 `Face-V2.0/data/` 里的 `meta.json` 和 `embeddings.pkl`，迁移成本就从“重建一套”降成了“拷个目录”。

从工程思维上说，这一轮最值得学的，不是某个函数怎么写，而是怎么判断“该删什么”。很多项目会越做越胖，不是因为必须胖，而是因为没人愿意承认有些东西只是历史包袱。好工程师会区分“业务真正依赖的能力”和“曾经为了演示、调试、验证而长出来的附属器官”。前者要保护，后者要有勇气切掉。

如果你以后继续扩展这块人脸能力，我建议按下面的顺序来，而不是一下子冲向大重构：

1. 先把本地运行时稳定住，确保业务活动图片能稳定回写 `main_person`。
2. 再决定要不要在 `mdams` 里补一个极简的人脸索引管理页面。
3. 最后才考虑是否把自动建索引、自动聚类、人工修订闭环做回平台。

一句话总结这个项目现在的样子：它像一座已经有主展厅、库房、申请窗口和讲解台的馆，AI 人脸识别不再是馆外请来的临时嘉宾，而是被安静地安排进了后台值班室。这样做不花哨，但很像真正会长期活下去的系统。
